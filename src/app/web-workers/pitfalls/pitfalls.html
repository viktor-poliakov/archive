Web Workers: нюансы и подводные камни

Воркеры мощные, но живут по своим правилам и не бесплатны. Соберём главное, о чём стоит помнить, прежде чем выносить работу в фоновый поток.

Нет доступа к DOM

Воркер не видит страницу: document, window и DOM ему недоступны. Значит, менять интерфейс из воркера нельзя — он только считает и присылает данные, а рисует главный поток:

Данные копируются — и это не бесплатно

Каждое postMessage делает копию данных (структурное клонирование). Для мелочи незаметно, но копировать десятки мегабайт — уже ощутимая цена. Для больших бинарных данных есть выход: передаваемые (Transferable) объекты — их не копируют, а «отдают» воркеру:

После передачи буфер в главном потоке становится пустым (byteLength равен 0) — владение перешло воркеру. Это осознанный компромисс: скорость вместо копии.

Что нельзя отправить

Через postMessage проходит не всё. Функции и DOM-узлы отправить нельзя — будет DataCloneError. А объекты копируются без методов — только данные:

Создание воркера не бесплатно

Запуск воркера требует времени и памяти (поднять поток, загрузить скрипт). Поэтому воркеры не создают на каждую мелкую задачу — выгода появляется на действительно тяжёлой работе. Если задач много, держат пул воркеров и переиспользуют их. И помните: файл воркера должен быть с того же origin, что и страница.

Кратко

  • Нет DOM/window: воркер только считает, интерфейс меняет главный поток.
  • Данные копируются при каждом сообщении; для больших буферов — Transferable (передача без копии).
  • Нельзя слать функции и DOM-узлы (DataCloneError); объекты теряют методы.
  • Старт воркера не бесплатен — не для мелких/частых задач; переиспользуйте через пул.
  • Только тот же origin для файла воркера.
