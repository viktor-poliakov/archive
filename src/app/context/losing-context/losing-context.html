Потеря контекста

Самая частая беда с this — он «теряется»: метод вроде бы тот же, а this внутри уже не тот объект. Разберём, когда это происходит и как чинить.

Почему this теряется

Причина всегда одна. this определяется по объекту перед точкой в момент вызова. Как только вызов превращается в простой fn() без точки — срабатывает привязка по умолчанию, и this становится undefined (в строгом режиме).

Потерять this нельзя «наполовину». Либо вы вызываете через точку (obj.method()) — и this равен объекту, либо вызов выродился в fn() — и работает привязка по умолчанию. Промежуточного не бывает.

  • сохранение метода в переменную;
  • деструктуризация метода;
  • передача метода в колбэк (setTimeout, forEach, addEventListener, then);
  • обработчики событий (особый случай);
  • вложенная обычная функция внутри метода.

Как сохранить контекст

Есть несколько приёмов — выбор зависит от ситуации:

Стрелка-обёртка

Оборачиваем вызов в стрелочную функцию и зовём метод через точку внутри неё — точка на месте, this корректный.

bind

method.bind(obj) возвращает новую функцию с навсегда привязанным this (см. «call, apply, bind»).

thisArg у forEach / map

Методы массивов принимают вторым аргументом объект для this. Работает только для обычных функций — стрелки его игнорируют.

const self = this (старый приём)

До стрелочных функций this сохраняли в обычную переменную (self / that) и обращались уже к ней. Сейчас почти не нужен, но встречается в старом коде.

Поле-стрелка в классе

В классах метод можно объявить как поле со стрелкой — тогда this привязан к экземпляру и не теряется даже при отрыве (см. «this в классах и new»).

Приём Когда уместен Минусы
Стрелка-обёртка разовая передача в колбэк создаётся новая функция при каждом вызове
bind нужна готовая привязанная функция тоже новая функция; легко забыть
thisArg методы массивов с обычным колбэком не работает со стрелками
поле-стрелка в классе обработчики в классах/компонентах своя функция на каждый экземпляр
