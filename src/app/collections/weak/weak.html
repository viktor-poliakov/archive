WeakMap и WeakSet

У Map и Set есть «слабые» родственники — WeakMap и WeakSet. Главное отличие: они держат объекты слабо, то есть не мешают сборщику мусора их удалить. За это платят урезанным набором возможностей.

Чтобы понять, зачем это нужно, полезно помнить про достижимость: обычный Map держит свои ключи-объекты, поэтому они не удаляются, пока жив сам Map. Слабая коллекция так не делает.

Сильная ссылка против слабой

WeakMap

WeakMap — как Map, но с двумя ограничениями: ключами могут быть только объекты, и ссылки на них слабые. Поэтому и API урезан: только get, set, has, delete — без перебора, size и clear:

Где применяют: кэш и метаданные «по объекту» — данные живут ровно столько, сколько жив сам объект, и чистить вручную не нужно (подробнее про память — в разделе «Утечки памяти»).

WeakSet

WeakSet — как Set, но хранит только объекты и тоже слабо; доступны лишь add, has, delete. Удобен как «пометка да/нет» на объектах — посещён, обработан, прочитан:

Почему их нельзя перебрать

У слабых коллекций нет keys(), size и цикла for..of. Причина в сборщике мусора: момент удаления недетерминирован, объект может исчезнуть в любой миг. Поэтому «текущее содержимое» невозможно надёжно показать — язык и не даёт такой возможности.

Когда что брать

  Map / Set WeakMap / WeakSet
Ключи / значения любые только объекты
Удержание объекта держит (мешает сборке) слабо (не мешает)
Перебор, size есть нет
Для чего хранение и обход данных данные «по объекту» на время его жизни

Кратко

  • WeakMap/WeakSet держат только объекты и только слабо.
  • Не мешают сборке: исчез объект — исчезла и запись, чистить вручную не нужно.
  • Нет перебора, size, clear — из-за недетерминированной сборки.
  • Применение: кэш/метаданные и пометки «по объекту».
  • Нужен обход или примитивные ключи — берите обычные Map/Set.
