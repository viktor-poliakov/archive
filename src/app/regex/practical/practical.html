Практика и подводные камни

Соберём изученное на реальных задачах: валидация, извлечение данных, замена и форматирование. А в конце — типичные грабли, о которых стоит знать заранее.

Валидация

Проверка «подходит ли строка под формат» — самое частое применение regex. Приём один: обрамляем шаблон якорями ^…$, чтобы строка совпала целиком. Для пароля пригодятся опережающие проверки (?=…) — «где-то есть буква», «где-то есть цифра».

Про email: «идеального» regex для email не существует — стандарт слишком сложный. Для формы достаточно простой проверки (как выше) плюс реальная отправка письма с подтверждением. Строгую валидацию полей удобно оформлять своими ошибками — см. Свои классы ошибок.

Извлечение данных

Вытащить из текста все совпадения — задача для matchAll с группой: хештеги, цены, ссылки, номера. Собираем результат в массив и берём нужную группу.

Замена и форматирование

Регулярка + replace отлично чистит и переформатирует текст: схлопнуть пробелы, переставить части, замаскировать данные. В маскировке карты помогает опережающая проверка «после этой цифры есть ещё минимум четыре».

Подводные камни

  • Жадность по умолчанию — .+ хватает лишнее; помните про ленивое .+? (см. Кванторы).
  • Флаг g хранит lastIndex — не переиспользуйте одну и ту же /g-регулярку в test подряд.
  • Экранируйте ввод пользователя перед вставкой в new RegExp — иначе его спецсимволы «оживут».
  • Катастрофический откат (ReDoS) — вложенные кванторы вроде (a+)+ на длинной строке могут «повесить» движок; избегайте их.
  • Не парсите HTML/JSON регуляркой — для вложенных структур нужны настоящие парсеры, regex тут бессилен.

Кратко

  • Валидация: ^…$ + при нужде (?=…); для email — простая проверка, не «идеальная».
  • Извлечение: matchAll + группа + сбор в массив.
  • Замена: replace с группами $1 или функцией.
  • Грабли: жадность, lastIndex у /g, неэкранированный ввод, ReDoS, «HTML регуляркой не парсят».
