HTTP-методы

В каждом запросе к API вы указываете не только с чем работаете (адрес ресурса, например /users/42), но и что именно хотите с ним сделать. Это «что именно» и есть HTTP-метод — глагол, который ставится перед ресурсом-существительным. Один и тот же адрес /users/42 можно прочитать (GET), обновить (PUT) или удалить (DELETE) — разница только в методе.

Аналогия — заявка в картотеку. Адрес — это карточка в ящике: «сотрудник №42». А метод — пометка на заявке: «показать карточку», «завести новую», «переписать целиком», «выбросить». Клерк смотрит на глагол и точно знает, что делать. Вы сами понимаете, что произойдёт, — здесь нет магии, правила методов чётко описаны стандартом.

Методы — глаголы над ресурсами

В приложениях данные почти всегда проходят четыре базовые операции: создать, прочитать, обновить, удалить. Их сокращённо называют CRUD (Create, Read, Update, Delete). REST кладёт на эти операции подходящие HTTP-методы — так, чтобы глагол читался естественно:

  • Create (создать) → POST;
  • Read (прочитать) → GET;
  • Update (обновить) → PUT (целиком) или PATCH (частично);
  • Delete (удалить) → DELETE.

Важная идея REST: адрес описывает ресурс (существительное), а метод — действие (глагол). Поэтому в правильном API нет URL вида /getUser или /deleteUser?id=42 — глагол не прячут в адрес, для него есть отдельное поле. Есть только ресурс /users/42, а что с ним делать, говорит метод.

Разбор методов

GET — получить

Читает ресурс и возвращает его представление. Тела запроса у GET нет: всё, что нужно серверу, передаётся в самом адресе (а фильтры и постраничность — в строке запроса, например /users?role=admin&page=2). GET ничего не меняет на сервере, поэтому его безопасно повторять и кешировать.

POST — создать

Отправляет данные на сервер, чтобы создать новый ресурс. Данные едут в теле запроса, а тип тела указывается заголовком Content-Type (обычно application/json). Запрос идёт на коллекцию (/users), потому что нового элемента ещё нет и его ID назначит сам сервер. Успешный ответ — код 201 Created, часто с заголовком Location, где лежит адрес созданного ресурса.

PUT — заменить целиком

Кладёт по указанному адресу полное новое представление ресурса, заменяя предыдущее целиком. Запрос идёт на конкретный элемент (/users/42), а в теле — весь объект. Всё, что вы не прислали, считается удалённым.

PATCH — обновить частично

Применяет к ресурсу частичные изменения: в теле — только те поля, которые нужно поменять. Остальное сервер оставляет как было. Удобно, когда меняется одно-два поля большого объекта.

DELETE — удалить

Удаляет ресурс по адресу. Тела запроса обычно нет. В ответ, как правило, приходит 200 OK (иногда с телом об удалённом объекте) или 204 No Content, если возвращать нечего.

curl — это консольная утилита: программа, которую запускают в терминале (командной строке), а не в браузере. Безопасность и идемпотентность

У методов есть два свойства, которые определяют, насколько спокойно их можно повторять. Это ключ к устойчивости приложения при сбоях сети.

  • Безопасный (safe) — метод только читает и не меняет состояние на сервере. Таковы GET и HEAD. Их можно вызывать сколько угодно раз без последствий.
  • Идемпотентный (idempotent) — повтор запроса приводит сервер к тому же итоговому состоянию, что и один запрос. Идемпотентны GET, PUT и DELETE; POST — нет; PATCH — обычно нет.

Любой безопасный метод автоматически идемпотентен (раз он ничего не меняет, то и повторять безопасно), но не наоборот: PUT и DELETE меняют состояние, но остаются идемпотентными.

Аналогия: кнопка лифта против корзины

Идемпотентно — как кнопка «этаж 3» в лифте: нажмите её один раз или десять, результат один — лифт едет на третий этаж. Таковы PUT («сделай role равным user» — сколько ни повторяй, значение одно) и DELETE («пусть ресурса не будет» — после первого раза его уже нет, повтор ничего не ломает).

Не идемпотентно — как кнопка «добавить в корзину»: каждое нажатие добавляет ещё один товар. Таков POST: два одинаковых запроса создадут два ресурса.

Метод Назначение (CRUD) Тело запроса Безопасный Идемпотентный
GET Read — прочитать нет да да
POST Create — создать да нет нет
PUT Update — заменить целиком да нет да
PATCH Update — обновить частично да нет обычно нет
DELETE Delete — удалить нет нет да

Зачем это на практике: если ответ на запрос потерялся в сети, клиент не знает, дошёл он или нет. Идемпотентный запрос (GET, PUT, DELETE) можно спокойно повторить — хуже не станет. А вот POST повторять опасно: рискуете создать дубль. Как безопасно повторять запросы и обрабатывать такие ситуации — на странице Обработка ошибок.

PUT против PATCH

Оба метода обновляют существующий ресурс, но по-разному. PUT требует полное представление и заменяет ресурс целиком: поля, которых нет в теле, будут стёрты. PATCH присылает только изменения и оставляет остальное нетронутым. Сравните на одном и том же исходном объекте:

Отсюда и разница в идемпотентности из таблицы выше: повторный PUT с тем же телом снова выставит те же поля — итог не изменится. А PATCH может быть неидемпотентным: например, инструкция «увеличить счётчик на 1» при каждом повторе даёт новый результат.

В браузере все эти запросы отправляются через fetch — как в примерах с методами выше. Полностью (заголовки, разбор ответа, асинхронность) это разобрано в разделах HTTP: запрос и ответ и Fetch API на практике.

Почему запрос асинхронный: сеть работает медленно и непредсказуемо, поэтому fetch возвращает промис, а не результат сразу — иначе интерфейс замер бы на время ожидания. Почему так устроено — в разделе Событийный цикл, а про передачу функций «на потом» — в Функции как значения.
