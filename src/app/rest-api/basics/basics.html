REST API: основы

Почти любое приложение сегодня разговаривает с сервером: лента новостей, корзина в магазине, список задач. Чтобы этот разговор был предсказуемым, нужен общий язык — API. А REST — это самый распространённый стиль такого языка для веба: набор простых договорённостей о том, как клиент и сервер обмениваются данными поверх HTTP.

Аналогия — ресторан. Вы (клиент) не идёте на кухню и не трогаете плиту. Вы читаете меню и делаете стандартный заказ официанту, а тот приносит готовое блюдо. Меню и правила заказа — это и есть API: договор, по которому две стороны понимают друг друга. Здесь нет «чёрного ящика» — ниже мы разберём, из каких именно правил этот договор состоит.

Что такое API и зачем он

API (Application Programming Interface) — это описанный заранее интерфейс, через который одна программа обращается к другой. Он говорит: «вот такие запросы я принимаю, вот в таком виде отвечаю». Вызывающей стороне не нужно знать, как всё устроено внутри — какая база данных, на каком языке написан сервер. Достаточно соблюдать договор.

Когда такой интерфейс доступен по сети и работает поверх HTTP, его называют Web API. Браузер, мобильное приложение и другой сервер — все они могут быть клиентами одного и того же Web API. Это как единое меню, по которому обслуживают любого посетителя.

Коротко: API — это договор между программами. REST — один из популярных способов оформить такой договор для веба.

Что значит REST

REST расшифровывается как REpresentational State Transfer — «передача представления состояния». Ключевое: это архитектурный стиль, а не протокол и не стандарт. REST ничего не заставляет — он лишь предлагает набор принципов, которым API может следовать.

Главная идея умещается в трёх пунктах:

  • всё, чем управляет система, — это ресурсы (пользователь, заказ, статья);
  • у каждого ресурса есть свой адрес — URL (/users/42);
  • действия над ресурсами выражают стандартными HTTP-методами (GET, POST, PUT, DELETE) — см. HTTP-методы.

API, который придерживается этих принципов, называют RESTful. Термин нестрогий: почти никто не следует всем правилам REST в оригинальном смысле, поэтому «RESTful» обычно значит просто «сделан в духе REST».

Ресурсы и представления

Ресурс — это любая сущность, о которой можно говорить: конкретный пользователь, список заказов, одна статья. Ресурс адресуется URL-ом. Принято, что коллекция — это множественное число (/users), а отдельный элемент — с идентификатором (/users/42).

Сам объект на сервере (строка в базе данных) наружу не отдаётся. Клиент получает представление (representation) этого ресурса — его снимок в удобном формате. Чаще всего это JSON. Вот как может выглядеть представление пользователя /users/42:

Как это выглядит: клиент и сервер

Схема любого обращения одинакова: клиент шлёт HTTP-запрос на URL ресурса, сервер выполняет работу (например, читает базу данных) и возвращает HTTP-ответ с представлением ресурса и кодом состояния.

REST ≠ HTTP: REST — это идея, а HTTP — конкретный протокол. Теоретически REST можно реализовать и поверх другого протокола, но на практике REST API почти всегда работает именно поверх HTTP. Как устроены сами запрос и ответ — на странице HTTP: запрос и ответ.

Принципы REST

REST держится на нескольких ограничениях. Разберём главные — те, что реально влияют на то, как вы пишете и потребляете API:

Клиент-сервер

Ответственность разделена: клиент отвечает за интерфейс и удобство пользователя, сервер — за данные и бизнес-логику. Они общаются только через API, поэтому их можно менять и развивать независимо друг от друга.

Stateless — без состояния

Каждый запрос самодостаточен: он несёт всё, что нужно серверу, чтобы его обработать (адрес ресурса, метод, токен авторизации, тело). Сервер не хранит состояние клиента между запросами — он не «помнит», что вы делали в прошлый раз.

Аналогия: каждый раз вы заново представляетесь и объясняете, чего хотите — как будто пришли к новому официанту. Это кажется избыточным, зато любой сервер из кластера может обработать любой запрос. Именно поэтому stateless-подход так важен для масштабирования. Почему при этом всё работает асинхронно и не блокирует интерфейс — см. Event Loop: основы.

Единообразный интерфейс

Все ресурсы адресуются по одинаковым правилам (URL), а действия выражаются одним и тем же набором HTTP-методов. Узнав, как устроен один эндпоинт, вы интуитивно понимаете и остальные — не нужно учить новый способ на каждый случай.

Кешируемость

Ответы могут помечаться как кешируемые. Тогда клиент или промежуточный узел сохранит результат и не будет дёргать сервер повторно за теми же данными — это ускоряет работу и снижает нагрузку. Управляют этим специальные заголовки.

Первый запрос

В браузере запросы к REST API делают через Fetch API. Вот минимальный пример: получаем представление пользователя по его URL и разбираем JSON-ответ в обычный объект JavaScript.

fetch возвращает промис, поэтому используем await — сама функция fetch здесь принимает URL как значение, а разбор ответа мы вызываем отдельным шагом (о функциях как значениях — на странице Функции как значения).

Осторожно: fetch считает запрос успешным даже при кодах 404 или 500 — промис отклоняется только при сетевом сбое. Поэтому статус проверяют вручную через response.ok. Все детали fetch, заголовков и обработки ошибок — на странице Fetch API на практике.
