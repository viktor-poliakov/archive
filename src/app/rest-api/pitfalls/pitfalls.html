Проектирование REST API и подводные камни

Спроектировать REST API — это как составить понятную инструкцию к прибору: другой человек должен угадывать, «за какую ручку тянуть», не заглядывая в исходники. Если адреса называются по одному правилу, ошибки приходят в едином виде, а большие списки не валятся на клиента целиком — с таким API приятно работать. Здесь разберём приёмы, которые делают API предсказуемым, и типичные ошибки, из-за которых он превращается в загадку.

Важная мысль: почти всё это — не про «уметь писать код», а про договорённости. Хорошее API — это набор понятных правил, которые вы держите одинаковыми во всех уголках сервиса.

Именование ресурсов

В основе REST лежит идея ресурса — сущности, к которой мы обращаемся: пользователь, заказ, товар. URL называет ресурс, а не действие над ним. Поэтому в пути стоит существительное во множественном числе (/users), а что с ним делать — говорит HTTP-метод (GET, POST и т. д.).

Глаголу в URL места нет: /getUser и /createUser — это дублирование того, что и так выражает метод. Один и тот же адрес /users/42 обслуживает разные действия в зависимости от метода: GET — прочитать, PATCH — изменить, DELETE — удалить.

  • Множественное число — коллекция это /users, а один элемент — /users/42.
  • Иерархия — вложенность показывает принадлежность: /users/42/orders — «заказы этого пользователя».
  • Никаких глаголов — действие всегда несёт метод, а не путь.

Версионирование

Рано или поздно формат ответа придётся менять: переименовать поле, убрать лишнее, изменить структуру. Но у API уже есть клиенты — мобильные приложения, чужие интеграции, — которые обновятся не сразу. Если сломать старый формат, сломаются и они. Версионирование позволяет выпустить новый формат, не трогая старый.

Чаще всего версию кладут прямо в путь (/v1/…, /v2/…): это видно в логах, легко тестировать вручную и удобно кэшировать. Реже версию передают заголовком Accept. Главное — завести версию с самого начала, чтобы потом не ломать существующих клиентов.

Пагинация, фильтрация, сортировка

Коллекция может содержать миллионы записей. Отдавать всё разом нельзя: ответ будет огромным, сервер и сеть перегрузятся, а клиенту столько и не нужно. Поэтому список нарезают на страницы и позволяют его сузить — всё это задаётся query-параметрами (страница, лимит, фильтры, сортировка). Как их правильно собирать и кодировать — на отдельной странице.

С точки зрения проектирования важнее, как устроен ответ. Сами записи обычно кладут в поле data, а рядом — метаданные о странице, чтобы клиент знал, есть ли ещё:

Единый формат ошибок

Когда что-то пошло не так, клиенту мало кода состояния — нужны детали: что именно сломалось и можно ли это показать пользователю. Худший вариант — когда каждый эндпоинт отвечает на ошибку по-своему: где-то строка, где-то массив, где-то поле msg, а где-то error. Клиенту приходится разбирать каждый случай отдельно.

Идемпотентность и повторы

Идемпотентный запрос можно повторить сколько угодно раз с тем же результатом. GET, PUT, DELETE — идемпотентны; повторный GET ничего не портит, а повторный DELETE оставляет ресурс удалённым. А вот POST не идемпотентен: каждый вызов создаёт новую запись.

Отсюда проблема: клиент отправил POST /orders, соединение оборвалось до ответа — и он не знает, дошёл заказ или нет. Повторит — рискует создать дубль. Приём Idempotency-Key решает это: клиент генерирует уникальный ключ один раз и шлёт его при каждом повторе. Сервер по ключу понимает, что это тот же запрос, и не выполняет операцию дважды.

REST — это соглашения, а не жёсткий стандарт. Ни один документ не обязывает вас именовать пути именно так или возвращать ошибки в таком виде. Все эти правила служат одной цели — предсказуемости и единообразию: чтобы, увидев одну часть API, разработчик уже угадывал, как устроена остальная.

Частые ошибки и как правильно

Большинство проблем повторяются от проекта к проекту. Вот сводка типичных ошибок и того, как стоит делать вместо них:

Частая ошибка Как правильно
Глагол в URL: POST /createUser Существительное + метод: POST /users
Всегда возвращать 200 OK, даже на ошибку Верный код: 201 при создании, 400/ 404/500 при ошибках
Отдавать всю коллекцию одним ответом Пагинация: GET /users?page=1&limit=20
Повторять неидемпотентный POST вслепую Слать заголовок Idempotency-Key и повторять с ним
Секретный ключ (API key) зашит во фронтенде Секреты держать на сервере, клиент ходит через свой бэкенд

Про правильные коды подробно — на странице Коды состояния, а про то, почему секреты нельзя держать в браузере, — в разделе Аутентификация.

Чеклист хорошего REST API

Короткий список для самопроверки перед тем, как отдавать API клиентам:

  • Пути — существительные во множественном числе, без глаголов (/users, не /getUsers).
  • Действие задаёт метод, а не URL; методы используются по назначению (GET не меняет данные).
  • Ответы несут корректные коды состояния, а не всегда 200.
  • Ошибки приходят в едином формате со стабильными полями (code, message).
  • Коллекции пагинируются; фильтр и сортировка — через query-параметры.
  • Есть версия (/v1/…), чтобы менять формат, не ломая клиентов.
  • Указаны правильные заголовки (Content-Type, Accept).
  • Секреты и ключи живут на сервере, а не во фронтенде (см. Аутентификация).
