Коды состояния HTTP

Когда клиент отправляет запрос, сервер обязан коротко сообщить, чем всё кончилось. Делает он это кодом состояния — трёхзначным числом в самой первой строке ответа: 200, 404, 500. Это язык, на котором сервер и клиент договорились заранее: по одному числу понятно, сработал запрос или нет и на чьей стороне проблема.

Аналогия — ответ курьера, который отвёз вашу посылку. Он не пересказывает всю поездку, а отвечает коротко: «доставлено», «адресат не найден», «нужен пропуск», «на складе авария». Код состояния — ровно такая пометка: по ней вы сразу понимаете, что делать дальше, не разбирая тело ответа целиком.

Клиенту не нужно угадывать результат по телу ответа: сначала он читает число и уже по нему выбирает ветку логики. Показать данные? Повторить запрос? Отправить пользователя на страницу входа? Ответ определяется в первую очередь кодом. Как из этого числа складывается вся стартовая строка ответа, разобрано на странице HTTP: запрос и ответ.

Главная подсказка — первая цифра. Она задаёт класс ответа. Даже незнакомый код вроде 418 можно грубо истолковать: 4xx — значит, ошибка на стороне клиента.

Пять семейств

Все коды делятся на пять групп по первой цифре. Запомнив смысл каждой группы, вы читаете большинство ответов не глядя в справочник:

  • 1xx — информационные. Промежуточный сигнал «запрос принят, продолжаю обработку». На практике в браузерном коде вы их почти не видите — протокол обрабатывает их сам.
  • 2xx — успех. Запрос принят и выполнен. Самый желанный класс: данные пришли, ресурс создан или удалён.
  • 3xx — перенаправление. Ресурс есть, но находится в другом месте или не изменился. Часто требует ещё одного шага (перейти по новому адресу или взять данные из кэша).
  • 4xx — ошибка клиента. Сервер понял запрос, но выполнять отказывается: неверные данные, нет прав, нет такого ресурса. Повторять тот же запрос без изменений бессмысленно.
  • 5xx — ошибка сервера. Запрос был корректным, но сервер сам не справился: сбой, перегрузка, недоступный сервис. Клиент часто может повторить попытку позже.

Ключевые коды на практике

Кодов в стандарте десятки, но в реальной работе с HTTP-методами и API вы постоянно встречаете один и тот же набор. Вот шпаргалка по самым важным — что означает код, когда он приходит и как на него реагировать клиенту:

Код Значение Когда возникает Что делать клиенту
200 OK Успех, есть тело ответа Удачный GET; успешный PUT/PATCH с телом Прочитать тело (обычно response.json())
201 Created Создан новый ресурс Успешный POST, который создал запись Взять адрес нового ресурса из заголовка Location
204 No Content Успех, тела нет Успешный DELETE или PUT без ответа Не вызывать .json() — тело пустое
301 / 302 Перемещён (навсегда / временно) Ресурс переехал на другой адрес Перейти по Location; fetch следует за редиректом сам
304 Not Modified Не изменился с прошлого раза Условный GET с If-None-Match/If-Modified-Since Взять данные из кэша, тело не придёт
400 Bad Request Некорректный запрос Битый JSON, отсутствует обязательное поле Исправить запрос; повтор «как есть» не поможет
401 Unauthorized Не аутентифицирован Нет токена или он истёк Войти заново, обновить токен (см. ниже)
403 Forbidden Нет прав на действие Пользователь известен, но доступ запрещён Не повторять; показать «нет доступа»
404 Not Found Ресурс не найден Неверный URL или запись удалена Проверить адрес; показать «не найдено»
405 Method Not Allowed Метод не поддержан DELETE там, где разрешён только GET Использовать метод из заголовка Allow
409 Conflict Конфликт состояния Дубликат записи, параллельное редактирование Перечитать данные и решить конфликт
422 Unprocessable Entity Синтаксис верен, данные — нет Провалилась бизнес-валидация полей Показать ошибки полей из тела ответа
429 Too Many Requests Слишком много запросов Превышен лимит частоты (rate limit) Подождать по заголовку Retry-After и повторить
500 Internal Server Error Сбой на сервере Необработанное исключение в коде сервера Повторить позже; сообщить об ошибке
502 / 503 Шлюз / сервис недоступен Перегрузка, деплой, недоступный upstream Повторить позже с задержкой (backoff)

Про 2xx и тело: 200 обычно несёт данные, а 204 и 304 — никогда. Вызывать у них response.json() нельзя: парсер упадёт на пустом теле. Сначала смотрите на код, потом решайте, читать ли тело.

401 против 403

Эти два кода путают чаще всего, потому что оба означают «отказано». Но причина отказа разная, и клиент должен реагировать по-разному:

401 — «Кто ты?» Сервер не понял, кто отправитель: токена нет, он просрочен или неверен. Лечится аутентификацией — войти заново или обновить токен.
403 — «Тебе нельзя.» Сервер прекрасно знает, кто вы, но на это действие у вас нет прав. Повторный вход не поможет — нужны другие полномочия. Подробнее — на странице Аутентификация.
