Подтипы и супертипы

Подтипы и супертипы — это про то, какое значение куда можно подставить. Если ухватить одну простую идею — «тип это множество значений» — вся система типов TypeScript станет предсказуемой: вы заранее будете понимать, где присваивание пройдёт, а где компилятор поставит красную волнистую линию.

Главная идея: тип — это множество значений

Мы привыкли думать про тип как про «ярлык»: тут строка, тут число. Но гораздо полезнее представлять тип как множество — набор всех значений, которые в этот тип «помещаются».

  • number — множество всех чисел: 0, 1, -7, 3.14, …
  • boolean — множество ровно из двух значений: true и false.
  • 'red' (литеральный тип) — множество из одного значения: только строка 'red'.
  • string | number — объединение множеств: все строки плюс все числа.

Аналогия. Тип — это коробка с этикеткой, а на этикетке написано, что именно в неё разрешено класть. В коробку «число» влезает любое число, в коробку «'red'» — только слово red и ничего больше. Чем строже этикетка, тем меньше вещей поместится.

Как только тип — это множество, слова «подтип» и «супертип» перестают быть страшными терминами. Это просто «поменьше множество» и «побольше множество».

Что значит «A — подтип B»

Определение простое: тип A — подтип типа B, если любое значение типа A можно без опаски подставить туда, где ждут B. Говорят ещё, что «A присваивается B» (по-английски — assignable, отсюда и текст ошибок компилятора со словом assignable).

В терминах множеств это значит: подтип — это подмножество (более узкое, конкретное), а супертип — надмножество (более широкое, общее). Всё множество A целиком лежит внутри множества B.

Почему это безопасно? Функция или переменная типа Animal умеет пользоваться только свойством name. У Dog оно тоже есть — значит, любая работа с Animal на объекте-собаке пройдёт гладко. Лишнее свойство breed никому не мешает.

Аналогия: биологическая таксономия. «Такса» — подтип «собаки», «собака» — подтип «животного». Где нужна «собака» — такса подойдёт всегда (она и есть собака). А наоборот нет: там, где нужна именно «такса», произвольная собака не годится — вдруг это овчарка. Другая житейская картинка — матрёшки: маленькая целиком помещается в большую, но большая в маленькую — нет.

Иерархия типов: от never до unknown

Все типы выстраиваются в иерархию «уже ↔ шире». Наверху — самый широкий тип unknown (в него влезает вообще всё), внизу — самый узкий never (пустое множество, в нём нет ни одного значения).

Двигаться по этой лестнице снизу вверх (от подтипа к супертипу) присваивание разрешает всегда:

Направление присваивания: вверх можно, вниз нельзя

Это самое частое место, где новичок спотыкается. Правило одно:

Присваивание Что происходит Разрешено?
подтип → супертип кладём узкое в широкое (расширение) ✅ всегда безопасно
супертип → подтип кладём широкое в узкое (сужение) ❌ нужна проверка

Почему вниз нельзя? Потому что в супертипе могут лежать значения, которых нет в подтипе — и компилятор не может гарантировать, что конкретно сейчас там подходящее значение:

Аналогия. «Где нужен фрукт — яблоко сойдёт» (подтип → супертип). Но «где нужно яблоко — не всякий фрукт подойдёт»: вдруг вам протянут банан (супертип → подтип). Чтобы двигаться вниз, значение надо сначала проверить — убедиться, что это правда яблоко.

То же самое — с аргументами функций

Передача аргумента — это обычное присваивание: значение «кладётся» в параметр. Поэтому и правило то же самое. Если параметр функции — супертип, в него можно передать любой подтип. А если параметр — подтип, супертип туда уже не влезет.

Вложенные множества: круги Эйлера

Ту же иерархию удобно нарисовать не лесенкой, а вложенными кругами. Круг внутри — это подмножество, то есть подтип. Внешний круг — надмножество, супертип.

Из этой картинки сразу видны два практических правила. Литерал — подтип своего примитива: 'red' целиком лежит внутри string. Подробнее о литеральных типах — на странице Литеральные типы.

И зеркальное правило: объединение — супертип каждого своего члена. Тип string — подтип string | number, потому что множество строк — часть множества «строки или числа».

Вершина и низ: unknown и never

unknown — это top type, супертип абсолютно всех типов. В переменную unknown можно положить что угодно, но достать оттуда конкретный тип без проверки нельзя — иначе терялась бы вся безопасность.

never — противоположность, bottom type, подтип всех типов сразу. Это пустое множество: значения типа never не существует. Именно поэтому его «можно» присвоить куда угодно — присваивать-то на деле нечего, и правило нарушить невозможно.

Эта пара красиво ведёт себя и в объединениях — прямо как операции над множествами. never (пустое множество) в объединении просто исчезает, а unknown (множество «всё») — наоборот, поглощает остальных:

Аналогия. unknown — как ящик с надписью «здесь что-то есть»: положить можно любую вещь, но прежде чем пользоваться — надо заглянуть и убедиться, что именно. never — как ящик с надписью «сюда ничего нельзя положить»: он всегда пуст. Про эту пару типов есть отдельные страницы — any и unknown и void и never.

Структурная типизация: больше свойств — значит подтип

TypeScript сравнивает типы объектов не по имени, а по форме — это называют «утиной» типизацией: «если нечто крякает как утка, то это утка». И здесь прячется контринтуитивный момент.

Объект с бо́льшим числом свойств — это подтип объекта с меньшим (при совпадающих типах общих свойств). То есть { name: string; age: number } — подтип { name: string }.

Почему «больше свойств» = «уже множество»? Потому что каждое требование отсекает часть объектов. Условию «есть name» удовлетворяет уйма объектов. А условию «есть name и age» — уже меньше: те, у кого нет age, отпадают. Больше требований → меньше подходящих значений → более узкое множество → подтип.

На практике это и объясняет, почему функции, ждущей { name: string }, спокойно передают объект с лишними полями:

Подвох: лишние поля и объектные литералы

Есть важное исключение из предыдущего примера. Передать объект с лишними полями через переменную — можно. А вот записать такой объект литералом прямо в вызове — TypeScript не даст: включается excess property check (проверка на лишние свойства).

Логика такая: если вы прямо тут же создаёте объект и пишете в нём поле z, которого нет в целевом типе, — скорее всего это опечатка или недоразумение, и TypeScript вас предупреждает. А когда объект уже лежит в переменной, действует обычное структурное правило (подтип с лишними полями подходит). Подробный разбор этого и других сюрпризов — на странице Нюансы и подводные камни.

Коротко

  • Тип — это множество значений. Подтип — подмножество (уже), супертип — надмножество (шире).
  • A — подтип B, если значение A можно подставить всюду, где ждут B.
  • Подтип → супертип присваивается свободно; супертип → подтип — только после проверки (сужения). Тот же принцип — и при передаче аргументов в функцию.
  • Иерархия: never ⊂ литералы ⊂ примитивы ⊂ unknown. never — низ (подтип всех), unknown — вершина (супертип всех).
  • Литерал — подтип своего примитива; объединение — супертип каждого своего члена.
  • Структурная типизация: больше свойств = подтип. У объектных литералов дополнительно срабатывает excess property check.
