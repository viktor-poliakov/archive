TypeScript/Базовые типы/Вывод типов

Вывод типов

Аннотации типов в TypeScript нужны не везде. Очень часто компилятор сам догадывается о типе по значению — это и есть вывод типов (type inference). Разберём, как именно он рассуждает, чем отличается вывод для let и const, и где полагаться на автопилот не стоит.

Что такое вывод типов

Когда вы присваиваете переменной значение, TypeScript смотрит на это значение и записывает тип за вас. Отдельно указывать : number или : string не нужно — тип уже известен из того, что лежит справа.

Важно понять: вывод типа — это не «без типов». Тип есть, он строгий, просто написан не вашими руками, а компилятором. Всё, что запрещено для явно указанного number, запрещено и для выведенного.

Аналогия. TypeScript — как внимательный ассистент. Вы кладёте на стол число — он молча подписывает папку «числа», кладёте строку — «строки». Пока вы работаете с очевидными значениями, подписывать папки вручную незачем. Но там, где начинаются границы и договорённости (что функция принимает и отдаёт наружу), подпись лучше поставить самому — об этом в конце урока.

let и const: расширение типа

Один и тот же 3 выводится по-разному в зависимости от того, как объявлена переменная. Причина — можно ли её потом переприсвоить.

Для let TypeScript рассуждает так: «значение может смениться на другое число, поэтому фиксировать ровно 3 нельзя — возьму весь тип number». Это называется расширением (widening): литерал расширяется до примитива. Для const переприсвоения не будет никогда, значит можно оставить самый точный тип — литеральный (ровно 3).

Аналогия. const — как гравировка на металле: значение и его тип зафиксированы навсегда, можно доверять точной надписи «3». let — как ценник на грифельной доске: сегодня 3, завтра 10. Раз надпись меняется, магазин обещает лишь общее «здесь будет число» (number).

Объекты и массивы расширяются всегда

Внутри объекта свойства можно менять, даже если сам объект объявлен через const. Поэтому их типы тоже расширяются до примитивов. Чтобы получить точные литеральные типы, значение «замораживают» через as const.

Вывод типа возврата функции

Тип, который функция возвращает, TypeScript вычисляет по её return. Отдельно писать его не обязательно — компилятор «прогоняет» тело и складывает результат. Если return несколько и они разного типа, получится их объединение (union).

Это удобно для маленьких внутренних функций. Но у параметров вывода обычно нет: у них нет значения, из которого можно было бы догадаться о типе, — поэтому параметры почти всегда аннотируют вручную (см. последнюю секцию).

Контекстная типизация

Иногда тип берётся не из значения справа, а из контекста, куда выражение попадает. Самый частый случай — колбэк. В nums.map((n) => ...) вы не пишете тип n, но TS уже знает: nums — это number[], метод map передаёт в колбэк элемент массива, значит n — это number.

Лучший общий тип массива

Когда вы пишете массив из разных значений, TypeScript не сдаётся, а ищет лучший общий тип — самый узкий тип, под который подходят все элементы. Для [1, 2, 3] это number[], а для [1, "a"] — объединение (string | number)[].

Когда типы всё-таки пишут руками

Вывод хорош для локальных значений, но у него есть слабые места. Есть три ситуации, где явная аннотация обязательна или просто разумна.

1. Сигнатуры публичных функций — фиксируем контракт

Для функции, которой пользуются другие модули, аннотируйте и параметры, и тип возврата. Тогда, если тело случайно начнёт возвращать не то, ошибка укажет прямо на функцию, а не всплывёт где-то у вызывающего кода.

2. Пустой массив — иначе получите any[]

Если элементов нет, TypeScript нечего анализировать, и тип получается any[] — проверки отключаются. Укажите тип элемента сразу.

3. Параметры без контекста — иначе неявный any

У обычного параметра нет значения, из которого можно вывести тип. Под strict это ошибка «implicit any». Аннотируйте параметр — а вот возврат чаще всего можно доверить выводу.

  • Публичные функции — аннотируйте параметры и тип возврата: это договор, который вы фиксируете явно.
  • Пустые массивы и коллекции — задавайте тип элемента (const a: string[] = []), чтобы не свалиться в any[].
  • Параметры функций — почти всегда аннотируйте (кроме колбэков с контекстом и параметров со значением по умолчанию).
  • Не аннотируйте очевидное — const n = 5 лишняя подпись : number только шумит.

Правило большого пальца. Внутри функции доверяйте выводу — он делает код короче и не отстаёт от изменений. На границах модуля (что входит и что выходит) подписывайте типы сами: там важна не краткость, а стабильный, проверяемый контракт.

Кратко

  • TS сам выводит типы из значений — аннотации нужны не везде.
  • Расширение: let x = 3 → number, const x = 3 → литерал 3. Объекты и массивы расширяются даже под const (точность даёт as const).
  • Возврат функции выводится по return; разные return дают объединение.
  • Контекстная типизация: тип параметра колбэка (и объекта события) берётся из контекста вызова.
  • Лучший общий тип: [1, "a"] → (string | number)[].
  • Аннотируйте явно: публичные сигнатуры, пустые массивы, параметры без контекста — чтобы ловить ошибки раньше и не получить any.
