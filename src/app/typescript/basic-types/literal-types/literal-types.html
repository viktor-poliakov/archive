Литеральные типы

Обычно тип — это множество значений: string — все строки на свете, number — все числа. Литеральный тип сужает это множество до одного конкретного значения: не «какая-то строка», а именно 'red'; не «какое-то число», а именно 42. Само значение играет роль типа.

Аналогия — светофор. Тип string — это «любой цвет, какой захотите». А литеральный тип 'red' — это «ровно красный, и никакой другой». Для светофора вторая формулировка куда безопаснее: она физически не даёт зажечь «фиолетовый».

Что такое литеральный тип

Литеральный тип записывается точно так же, как само значение: строка в кавычках, число или true/false. Переменной с таким типом можно присвоить только это значение — любое другое TypeScript отвергнет ещё на этапе компиляции, не дожидаясь запуска программы.

Главный смысл — в объединении

Один литерал сам по себе нужен редко. Настоящая польза появляется, когда мы соединяем несколько литералов через | и получаем конечный список допустимых значений. Это надёжнее, чем string: набор вариантов зафиксирован, и ничего лишнего в переменную уже не попадёт.

Зачем это нужно. С обычным string опечатку никто не поймает: 'rihgt' — тоже валидная строка. С литеральным объединением редактор подсветит ошибку сразу и вдобавок подскажет автодополнением все допустимые варианты.

Литералы можно смешивать с примитивами

В одном объединении литералы и примитивы прекрасно уживаются. Классический пример — CSS-ширина: это любое число или одно ключевое слово 'auto'. Так мы разрешаем целый примитив number и добавляем к нему единственную допустимую строку.

Расширение типа (widening): let против const

Когда вы объявляете переменную без явной аннотации, TypeScript сам решает, какой тип вывести — и это решение зависит от того, let это или const:

  • const s = 'red' — значение никогда не изменится, поэтому TS сохраняет узкий литеральный тип 'red'.
  • let s = 'red' — значение можно переприсвоить, поэтому TS расширяет (widening) тип до примитива string.

Почему TS так делает

Логика простая: let заведён именно для того, чтобы значение потом менять. Если бы let s = 'red' имел тип 'red', то первая же попытка присвоить 'green' стала бы ошибкой — работать с такой переменной было бы невозможно. Поэтому TS заранее берёт тип пошире.

Заморозка через as const

С объектами и массивами возникает та же история, что и с let: их содержимое можно менять, поэтому TS выводит широкие типы (number, string[]). Чтобы поля стали литеральными и вдобавок readonly, к значению добавляют as const.

Аналогия — музейная витрина. Обычный объект — это вещь на столе: её можно переставить и заменить. as const ставит вещь под стекло: смотреть можно, а трогать и подменять — нет. Именно поэтому все поля становятся readonly, а их типы «схлопываются» до точных значений.

С массивами as const ещё полезнее: он превращает массив в readonly-кортеж из литералов, из которого затем удобно вывести объединение одной строкой.

Где это применяют на практике

Статусы приложения

Экран данных почти всегда находится в одном из нескольких состояний. Литеральное объединение описывает их точно, а switch с явным типом результата заставляет обработать каждое: если пропустить случай, TypeScript пожалуется, что функция не всегда что-то возвращает.

А чтобы TS сам напоминал про забытые ветки, когда список статусов вырастет, добавляют «страховку на исчерпание» через тип never. Пока обработаны все варианты, в default попасть нельзя, и присваивание в never проходит. Стоит добавить в объединение новый статус и забыть про него — эта строка загорится ошибкой.

Варианты и размеры компонента

В UI-компонентах — кнопках, бейджах, инпутах — набор допустимых вариантов конечен. Литеральные типы не дадут передать несуществующий variant или size и подскажут правильные значения прямо в редакторе.

Наборы кодов и методов

HTTP-методы, коды ответа, разрешённые действия — всё это фиксированные списки. И строковые, и числовые литералы одинаково хорошо для этого подходят.

Связь с подтипами

Литеральный тип — это всегда подтип своего примитива: 'red' — подтип string, 42 — подтип number, true — подтип boolean. Отсюда правило совместимости: значение с литеральным типом свободно кладётся туда, где ждут примитив, но не наоборот.

Почему «красный кладётся в цвет, а цвет в красный — нет», подробно разбираем в разделе Подтипы и супертипы.

Коротко

  • Литеральный тип — это конкретное значение ('red', 42, true) в роли типа.
  • Сила раскрывается в объединении: 'left' | 'right' — надёжный конечный список, безопаснее чем string.
  • Литералы можно смешивать с примитивами: number | 'auto'.
  • const сохраняет литеральный тип, let расширяет его до примитива (widening) — потому что let допускает переприсвоение.
  • as const замораживает объект/массив: поля становятся литеральными и readonly.
  • Литерал — это подтип своего примитива.
