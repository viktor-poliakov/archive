TypeScript/Базовые типы/void и never

void и never

Это два «крайних» типа TypeScript, связанных с возвратом из функций. void — «функция отработала, но осмысленного значения не вернула». never — «функция вообще не вернёт управление» или «такого значения просто не бывает». Их часто путают, хотя они противоположны: у void одно допустимое значение, а у never — ни одного.

Короткая аналогия. void — это работник, который сделал дело и молча ушёл: он вернулся, но отчёта не принёс. never — это дорога, которая никуда не ведёт: пойти по ней нельзя, конца у неё нет.

void: функция без осмысленного результата

Тип void ставят функции, которая работает ради побочного эффекта, а не ради значения: пишет в консоль, меняет DOM, отправляет запрос, обрабатывает событие. Возвращать ей нечего, поэтому её результат — void.

Почти каждый обработчик события и колбэк, который «просто что-то делает», имеет тип возврата void. Вы встречаете его постоянно: addEventListener, forEach, setTimeout — все ждут функцию, возвращающую void.

Аналогия. Вы просите коллегу «выключи свет в переговорке». Он идёт, выключает и возвращается. Спрашивать у него «ну и что ты принёс?» бессмысленно — задача была не про результат, а про действие. Вот это «вернулся с пустыми руками» и есть void.

void против undefined

Новички думают, что void и undefined — одно и то же. Значение у них правда совпадает (функция без return отдаёт undefined), но смысл разный:

  • void — «на моё возвращаемое значение не смотри, оно неважно». Можно вообще не писать return.
  • undefined — «я обязан вернуть значение, и это значение именно undefined». return обязателен.

Правило на практике. Для функций почти всегда пишут void, а undefined как тип возврата — редкость. Разницу проще запомнить так: void — про намерение («результата нет»), undefined — про конкретное значение.

Почему колбэк с типом void принимает функцию, что-то возвращающую

Самый неочевидный момент. Если тип колбэка объявлен как «верни void», в него разрешено передать функцию, которая на самом деле что-то возвращает. Это не баг, а специальное правило TypeScript.

Зачем так сделано? Без этого правила нельзя было бы написать привычное arr.forEach(x => list.push(x)): ведь push возвращает число, а forEach хочет void. Тип возврата void означает ровно «мне всё равно, что ты вернёшь — я это проигнорирую». Поэтому подойдёт функция с любым результатом.

Аналогия. Вы говорите курьеру: «просто оставь посылку у двери, сдачу не нужна». Если курьер всё-таки протянет сдачу — ничего страшного, вы её не берёте. Тип void — это то самое «сдачу не нужно»: вернуть могут что угодно, но брать вы не обязаны.

Обратная сторона: снаружи результат такой функции всё равно считается void, и использовать его нельзя.

never: тип без единого значения

never — это тип, в котором нет ни одного возможного значения. Переменной типа never нельзя ничего присвоить — просто нечего. Такой тип получают функции, которые никогда не возвращают управление: они либо всегда бросают исключение, либо крутятся в бесконечном цикле.

Ключевое отличие от void. void-функция завершается — просто ничего не принесла. never-функция не завершается: выполнение после её вызова не продолжится. Именно поэтому после вызова fail(...) можно не писать return — компилятор знает, что дальше кода уже не будет.

never в «невозможных» ветках

Второй способ встретить never — не объявлять его руками, а получить от компилятора при сужении типов. Когда TypeScript исключил все возможные варианты, у переменной не остаётся ни одного типа — и он становится never.

В блоке else значение уже не может быть ни строкой, ни числом — других вариантов у типа string | number нет. TypeScript отмечает такую ветку как never: «сюда исполнение не придёт». Это и есть основа приёма, который разберём дальше.

Практический приём: проверка полноты switch (exhaustiveness)

Представьте объединение вариантов (например, фигуры) и switch, который обрабатывает каждый. Хочется гарантии: если завтра добавят новый вариант, а обработать его забудут — сборка должна упасть. Ровно это и даёт never.

Фокус в ветке default: после того как разобраны все case, в default значение приходит с типом never. Присваиваем его в переменную типа never — пока вариантов не осталось, всё сходится.

А теперь добавим третий вариант triangle и «забудем» его обработать. Остаток в default перестаёт быть пустым — там оседает { kind: "triangle" }, и присвоить его в never уже нельзя. Сборка падает — то, что нам и нужно.

Аналогия. Это как контрольный список на выходе из дома: «плита, утюг, окна». Если каждый пункт вычеркнут — список пуст, можно уходить. Стоит добавить в список «кот», но не проверить его — список уже не пуст, и «уйти» не получится. Пустой список — это never, а непустой ловит компилятор.

never в операциях над типами

Раз у never нет значений, он ведёт себя как «нейтральный ноль» в объединениях: добавление never в | ничего не меняет. Отсюда и берётся пустой остаток в exhaustiveness-проверке.

Ещё одно свойство: never — подтип любого типа. Его можно присвоить куда угодно (const n: string = valueOfNever), но ему нельзя присвоить ничего. Именно поэтому assertNever с типом возврата never удобно подставлять в любую ветку — он совместим с любым ожидаемым результатом. Подробнее о направлении присваивания — в подтипах и супертипах.

Итог: void и never рядом

Соберём различия в одну таблицу, чтобы больше их не путать.

  void never
Сколько значений у типа одно — undefined ноль, ни одного
Функция завершается? да, но без результата нет, никогда не возвращает управление
Типичный пример console.log, обработчик события throw, бесконечный цикл
В объединении T | ... обычный член объединения исчезает: T | never = T
Присваивание принимает undefined ему нельзя присвоить ничего; сам присваивается любому типу
Аналогия работник сделал дело и молча ушёл дорога в никуда, «такого не бывает»
  • Ставьте void функциям ради побочного эффекта — а чаще просто доверьте вывод компилятору.
  • Помните правило колбэков: тип «верни void» принимает функцию с любым результатом, но снаружи результат считается void.
  • never редко пишут руками — обычно его возвращают функции-«кидалки» и получают в exhaustiveness-проверке.
  • Приём с const _: never = value в default — дешёвая страховка: забыли новый вариант — сборка падает.
