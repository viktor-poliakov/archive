TypeScript/Базовые типы/any и unknown

any и unknown

Иногда тип значения заранее неизвестен: пришли данные с сервера, значение прилетело из нетипизированной библиотеки, отработал JSON.parse. Для таких случаев в TypeScript есть два «универсальных» типа — any и unknown. Оба принимают в себя что угодно, но ведут себя противоположно: any отключает проверки и развязывает руки, unknown — наоборот, заставляет сначала убедиться, что за значение вам досталось. Разберём оба и поймём, почему по умолчанию выбирают unknown.

any — режим «делай что хочешь»

Тип any означает «я не знаю (или не хочу указывать) тип, отключи проверку». С переменной any разрешено вообще всё: присваивать любое значение, обращаться к любым свойствам, вызывать её как функцию, передавать куда угодно. Компилятор просто отходит в сторону.

Аналогия. any — это как поставить подпись на пустом бланке. Вы доверяете содержимому, не читая его: вдруг там всё в порядке. Пока данные и правда те, что вы ждёте, — работает. Но никакой защиты у вас нет.

По сути any возвращает вас в обычный нетипизированный JavaScript. Отсюда правило: any — это не «любой тип», а «выключить TypeScript в этом месте».

any «заражает» соседей

Опасность any не ограничивается одной переменной. Тип растекается: всё, что вычислено из значения any, тоже становится any. Достаточно одной такой переменной, чтобы проверки молча отключились в целой цепочке кода.

Аналогия. unknown — это запечатанная коробка с неизвестным содержимым. Держать её и передавать другому можно свободно. Но прежде чем использовать то, что внутри, коробку надо вскрыть и убедиться, что там именно то, что вы думаете. Никаких действий вслепую.

unknown называют «верхним» типом (top type): в него укладывается любое значение, поэтому он «сверху» над всеми. При этом сам он вниз, в конкретные типы, без проверки не проходит — в отличие от any, который совместим в обе стороны:

any vs unknown: что где разрешено

Оба типа принимают любое значение. Разница — в том, что компилятор позволяет делать дальше. Наглядно (зелёное — «да», красное — «нет»):

Можно ли… any unknown
присвоить в переменную любое значение да да
сразу обратиться к свойству / вызвать, без проверки да нет
присвоить напрямую в переменную конкретного типа да нет
компилятор ловит ошибки операций на этапе сборки нет да
обязательно сузить тип перед использованием нет да

Флаг noImplicitAny

any появляется не только когда вы пишете его руками. Если у параметра или переменной не удаётся вывести тип, TypeScript может подставить any неявно — молча. Флаг noImplicitAny запрещает это: любое «случайное» any становится ошибкой сборки, и вы обязаны указать тип явно.

Флаг входит в набор strict, поэтому в новых проектах он обычно уже включён:

Держите noImplicitAny включённым. Он превращает any из «того, что случается само» в осознанное решение, которое видно в коде.

Когда any всё-таки оправдан

any — не «зло», а инструмент с узким применением. Есть ситуации, где он уместен:

  • Быстрая миграция с JavaScript. Когда переводите большой JS-проект на TypeScript файл за файлом, any позволяет временно заглушить ошибки и не переписывать всё сразу.
  • Нетипизированная библиотека. Если у пакета нет описаний типов (.d.ts), проще временно объявить его как any, чем описывать типы целиком.
  • Прототип «на выброс». Когда важна скорость, а код всё равно скоро перепишут.

Во всех этих случаях any — осознанное, точечное исключение, а не стиль жизни. По умолчанию, когда тип заранее неизвестен, берите unknown: он ничего не ломает, но заставляет проверить значение перед использованием.

Кратко

  • any — «выключить TypeScript в этом месте»: разрешено всё, проверки нет, ошибки уходят в рантайм.
  • any заражает — всё, что из него вычислено, тоже становится any.
  • unknown — безопасная «вершина»: принимает любое значение, но использовать его нельзя, пока не сузишь тип.
  • Сужение — через typeof, instanceof, Array.isArray и сравнения; только после проверки значение доступно.
  • По умолчанию — unknown. any оставляйте для миграции и нетипизированных библиотек.
  • Включайте noImplicitAny (он входит в strict), чтобы any не появлялся сам собой.
