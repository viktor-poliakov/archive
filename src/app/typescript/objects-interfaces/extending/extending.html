Расширение и наследование

Большие типы почти никогда не пишут «с нуля» целиком. Их собирают из маленьких, как конструктор: берут готовый тип и добавляют к нему пару своих полей. В этом уроке — два инструмента для такой сборки: interface extends (расширение) и пересечение типов через оператор &. Оба избавляют от копипаста и делают код устойчивым к правкам.

Проблема: одни и те же поля снова и снова

Представим тип пользователя User и тип администратора Admin. Админ — это по сути тот же пользователь, но с дополнительным полем role. Если писать Admin целиком заново, придётся дословно повторить все поля User — и это плохо кончится.

Аналогия. Это как бумажная анкета. Есть базовый бланк с общими графами (имя, e-mail). Для админа не печатают новый бланк с нуля — берут тот же базовый и подкладывают доп. лист с графой «роль». Меняется общая графа — правят один базовый бланк, а не десяток его копий.

Повторять поля вручную Строить из кусочков (extends / &)
Копипаст всех полей в каждый новый тип Поле описано один раз и переиспользуется
Добавили поле — нужно править все копии Добавили в базовый — попадёт во все наследники
Копии со временем незаметно расходятся Один источник правды, рассинхрона нет

interface extends — расширяем интерфейс

Ключевое слово extends ставится после имени интерфейса и указывает, от кого «наследоваться». Наследник получает все поля родителя и добавляет свои. Это прямой аналог наследования классов, только для описаний данных (форм объектов).

Почему так. Раз Admin содержит всё, что есть в User, то он годится всюду, где ждут User. На языке системы типов Admin — подтип User. Подробнее про «что куда подставляется» — в уроке о базовых типах и в разделе объединений.

Множественное расширение

Один интерфейс можно расширить сразу несколькими — перечислите их через запятую. Так удобно собирать тип из независимых «черт»: у одной сущности есть идентификатор, у другой — даты создания и правки, у третьей — и то, и другое.

Важно. Если два родителя объявляют поле с одинаковым именем, но несовместимыми типами, TypeScript сообщит об ошибке — расширить такие интерфейсы сразу нельзя. При совпадающих типах конфликта нет.

Пересечение через type и оператор &

У псевдонимов type нет слова extends, зато есть оператор & — пересечение типов. Запись A & B означает «тип, у которого есть И всё из A, И всё из B». Для объектов результат тот же, что у extends: получается тип со всеми полями сразу.

Аналогия. Пересечение — это как список требований к кандидату на работу. «Знает TypeScript» & «знает Angular» означает, что подходит только тот, кто удовлетворяет обоим требованиям сразу. Чем больше условий соединяем через &, тем «богаче» должен быть объект — в нём должны быть все перечисленные поля.

interface extends type с &
Только для интерфейсов (форм объектов) Пересекает любые типы, не только объекты
Конфликт полей — сразу ошибка на объявлении Конфликт полей может дать тип never у поля
Привычно тем, кто пришёл из классов (ООП) Гибче: годится для union, функций, кортежей

Разницу между инструментами подробно разбираем на страницах Интерфейсы, Псевдонимы типов и interface или type.

interface и type прекрасно дружат

Их не нужно выбирать «на всю жизнь» — они свободно сочетаются. interface может расширять type-псевдоним объекта, а type — пересекать interface. Смешивайте так, как удобнее в конкретном месте.

Важно. interface умеет расширять type только тогда, когда тот описывает объект (или пересечение объектов). Расширить интерфейсом, например, объединение string | number нельзя — расширять там попросту нечего.

Переопределение поля в наследнике

В наследнике можно уточнить тип унаследованного поля — но только до совместимого, то есть до подтипа исходного. Сузить string до конкретного литерала 'circle' можно. А заменить number на string — нельзя: это уже не уточнение, а несовместимая подмена.

Почему так. Наследник обязан оставаться полноценной заменой родителя. Если код рассчитывает, что size — число, а наследник вдруг положит туда строку, такой объект уже не сможет притворяться родителем — контракт нарушен. Поэтому переопределять поле допустимо лишь на более узкий, совместимый тип.

Практика: цепочка и композиция

Соберём типичную иерархию из базы данных. Начинаем с крошечного Entity (у всего есть id), затем наращиваем: User расширяет Entity, а Admin расширяет User. Получается цепочка наследования Admin → User → Entity.

Второй подход — композиция: вместо длинной цепочки собирают тип из независимых «примесей» через &. Это как кубики LEGO: берём ровно те детали, что нужны, и не тянем лишнего.

Хотите глубже про необязательные и неизменяемые поля таких типов — загляните в Опциональные и readonly, а про типы-словари с произвольными ключами — в Индексные сигнатуры.

Кратко

  • Не повторяйте поля вручную — стройте большие типы из маленьких.
  • interface Admin extends User { role: string } — наследник получает все поля User плюс свои.
  • Множественное расширение: interface C extends A, B { … } — собрать тип из нескольких сразу.
  • Пересечение type Admin = User & { role: string } даёт тот же результат другим инструментом; & = «И то, И другое».
  • interface и type дружат: interface может extends от type-объекта, а type — пересекать interface.
  • Переопределять поле в наследнике можно только на совместимый (более узкий) тип; несовместимая замена — ошибка.
  • Наследник — подтип родителя: подходит всюду, где ждут родителя.
