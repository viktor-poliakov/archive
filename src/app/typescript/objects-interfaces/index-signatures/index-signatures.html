Индексные сигнатуры

Иногда мы заранее знаем все поля объекта поимённо: у пользователя есть name и age. Но бывает иначе — ключей может быть сколько угодно, а имена их наперёд неизвестны: словарь переводов, счётчики, кэш. Для таких объектов-словарей в TypeScript есть индексная сигнатура — запись вида { [key: string]: T }. Разберём её от самого простого к практике.

Проблема: ключи заранее неизвестны

Обычный тип объекта перечисляет свойства по именам. Это удобно, когда набор полей фиксирован. Но представьте словарь населения городов: ключи — названия городов, и их может быть сколько угодно. Перечислить каждый ключ в типе невозможно.

Аналогия. Индексная сигнатура — это шкаф с одинаковыми ячейками. Ячеек можно навесить сколько угодно, подписать их как захочется («Москва», «Казань», «носки», «варежки»), но во всех ячейках лежат вещи одного сорта. Тип говорит не «какие именно ячейки бывают», а «что лежит в любой ячейке».

Синтаксис

Индексная сигнатура пишется в квадратных скобках: [имяКлюча: string]: ТипЗначения. Её можно объявить и в интерфейсе, и в псевдониме типа — разницы в поведении нет.

Важно. Слово внутри квадратных скобок (city, key, город) — это просто имя-заглушка для читателя. Оно нигде не используется как настоящая переменная и не влияет на тип: [key: string] и [город: string] — один и тот же тип. Выбирайте имя так, чтобы оно подсказывало, что за ключ вы ожидаете.

Какими бывают ключи

Тип ключа может быть string, number или symbol. Чаще всего используют string — обычный словарь. Есть тонкость: в JavaScript у объекта не бывает настоящих числовых ключей — движок приводит число к строке, поэтому obj[0] и obj['0'] — один и тот же элемент. Из-за этого тип значения у числового ключа обязан быть совместим с типом значения у строкового.

Сочетание с известными свойствами

Часто у объекта есть и пара точно известных полей, и произвольные ключи. Индексную сигнатуру разрешено сочетать с обычными свойствами, но действует строгое правило: тип каждого явного свойства обязан быть совместим с типом значения индексной сигнатуры. Логика простая: раз «любой ключ даёт number», то и конкретный известный ключ тоже обязан давать number.

Явное свойство Тип значения сигнатуры Совместимо?
total: number [k: string]: number да — number подходит под number
version: string [k: string]: number нет — string не влезает в number
theme: string [k: string]: string | number да — string входит в string | number

А если явные свойства разного типа, расширьте тип значения сигнатуры до объединения (например, string | number), чтобы в него поместились все, — как в примере ниже.

Опасность: доступ может вернуть undefined

Вот самый коварный момент. Индексная сигнатура обещает, что под ключом лежит значение типа T. Но объект-словарь на то и словарь, что нужного ключа в нём может просто не оказаться. По умолчанию TypeScript верит сигнатуре на слово и молча выдаёт вам T — хотя в реальности там undefined.

Флаг noUncheckedIndexedAccess

Эту дыру закрывает флаг компилятора noUncheckedIndexedAccess в tsconfig.json. С ним доступ по ключу возвращает не T, а T | undefined — и компилятор заставит вас проверить значение перед использованием.

Почему так. Без этого флага система типов «уверена» в том, чего не знает, и ошибка всплывёт только в рантайме. Включённый noUncheckedIndexedAccess честно признаёт неопределённость и подталкивает к безопасным привычкам — проверке через if или значению по умолчанию через ??.

Короче и нагляднее: Record

Ту же идею словаря обычно выражают встроенной утилитой Record<Ключи, Значение>. Запись Record<string, number> — это ровно то же самое, что { [key: string]: number }, но короче и читается легче. Более того, Record позволяет ограничить ключи конкретным набором строк.

Когда ключи произвольные — берите индексную сигнатуру или Record<string, T>, как удобнее. Когда ключей фиксированный набор — Record с объединением строк даёт больше защиты. Подробный разбор — на странице Record.

Практика

Соберём три типичные задачи, где без словаря не обойтись.

Счётчик частоты слов

Ключи — сами слова, значения — сколько раз слово встретилось. Заранее слова неизвестны, поэтому словарь идеален.

Простой кэш ответов

Ключ — адрес запроса, значение — сохранённый ответ. Обратите внимание: функция чтения честно возвращает string | undefined — ведь в кэше может быть пусто.

Словарь переводов ru → en

Классический словарь: русское слово как ключ, английское как значение. На случай отсутствия перевода возвращаем исходное слово через ??.

Кратко

  • Индексная сигнатура { [key: string]: T } описывает объект-словарь, когда имена ключей заранее неизвестны.
  • Имя ключа в скобках — просто заглушка для читателя; на тип оно не влияет.
  • Тип ключа — string, number или symbol. Числовые ключи в JS всё равно приводятся к строкам.
  • Явные свойства обязаны быть совместимы с типом значения сигнатуры; иначе расширьте её тип до объединения.
  • Доступ по произвольному ключу может дать undefined. Флаг noUncheckedIndexedAccess добавляет | undefined и заставляет проверять.
  • Record<Ключи, Значение> — короткая и более гибкая альтернатива (см. Record).
