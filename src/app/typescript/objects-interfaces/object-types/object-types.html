Типы объектов

Объект в JavaScript — это набор именованных свойств: у него есть поля, а у каждого поля — своё значение. Тип объекта в TypeScript просто описывает эту форму: перечисляет, какие поля ожидаются и какого они типа. Самый прямой способ задать такую форму — написать её инлайном, прямо в аннотации: let user: { name: string; age: number }. С этого и начнём.

Что такое тип объекта

Представьте бумажный бланк-анкету: слева напечатаны подписи полей («Имя», «Возраст»), а справа — пустые строки, куда вписывают данные. У каждого поля свой смысл и свой «сорт» данных: в «Имя» пишут текст, в «Возраст» — число. Тип объекта в TypeScript — это и есть такой бланк: он говорит, какие поля должны быть и что в них класть.

Аналогия. Тип объекта — как пустой бланк, а конкретный объект — как заполненный экземпляр этого бланка. Бланк не хранит данные, он лишь задаёт форму: «здесь имя (текст), здесь возраст (число)». Пытаетесь вписать в графу «Возраст» слово вместо числа — TypeScript, как строгий регистратор, сразу вернёт бланк с пометкой об ошибке.

Обратите внимание: как только у переменной есть тип-форма, редактор знает про её поля. Он подсказывает name и age при вводе точки и сразу подчёркивает обращение к несуществующему полю — ещё до запуска программы.

Синтаксис: чем разделять свойства

Форма пишется в фигурных скобках, а свойства внутри — как пары имя: тип. Разделить их можно тремя равнозначными способами: точкой с запятой, запятой или просто переносом строки. Никакой разницы в смысле нет — выбирайте по вкусу (общепринятый стиль — точка с запятой).

Для опытных. Форматтер Prettier по умолчанию расставит здесь точки с запятой и разложит длинную форму по строкам. Так что чем именно разделять — обычно решает не человек, а автоформат; важно лишь помнить, что синтаксис допускает любой из вариантов.

Вложенные объекты

Значением поля может быть снова объект. Тогда и тип поля — снова тип объекта, вложенный в фигурные скобки. Получается «бланк внутри бланка»: например, у заказа есть поле customer, а оно само — объект с полями name и vip.

До вложенного поля добираются через точку, шаг за шагом: order.customer.name. TypeScript на каждом шаге знает форму и не даст свернуть «не туда».

Что именно проверяет TypeScript

Имея форму, компилятор сверяет с ней каждое присваивание. Три самые частые претензии: нет обязательного поля, поле не того типа и лишнее поле у объектного литерала. Разберём на одном типе-точке.

Лишние поля: только у литералов

С лишним полем есть тонкость. Если вы создаёте объект литералом прямо в месте присваивания и дописываете поле, которого нет в типе, — TypeScript посчитает это опечаткой и не пропустит (это называют excess property check). А вот если тот же объект сначала положить в переменную, а потом присвоить — лишнее поле пройдёт.

Важно. Это не противоречие, а намеренная защита от опечаток именно в литералах. Почему через переменную «лишнее» разрешено и как это связано со структурной типизацией — подробно на странице Нюансы и подводные камни.

Тип на переменной — это не само значение

Новичка легко сбивает то, что тип и значение объекта пишутся похоже — и там, и там фигурные скобки. Но роли у них разные. Слева от знака = стоит тип (описание формы), справа — значение (реальные данные).

Тип { size: number } Значение { size: 10 }
описывает форму: «есть поле size, оно число» конкретные данные: в size лежит 10
после компиляции исчезает — в рантайме его нет существует во время работы программы
в поле пишут тип (number) в поле пишут значение (10)

Почему так. Типы нужны только компилятору, чтобы поймать ошибки заранее. Когда TypeScript превращается в JavaScript, все аннотации стираются, и до браузера доедет лишь голое значение. Поэтому «поле типа number» и «поле со значением 10» — вещи из разных миров, хоть и стоят рядом.

Практика: объект в функциях

Чаще всего инлайновые типы объектов встречаются у функций — когда надо описать «пакет» из нескольких значений в одном параметре или вернуть сразу несколько полей.

Объект как параметр

Вместо длинного списка отдельных аргументов функции часто передают один объект с именованными полями. Тип такого параметра — обычный тип объекта.

Объект как возвращаемое значение

Функция может вернуть сразу несколько значений, упаковав их в объект. Форму результата описывают после круглых скобок — там же, где обычно ставят тип возврата.

Когда инлайн становится тесен

Инлайновые типы удобны, пока форма маленькая и используется в одном месте. Но стоит описать её в нескольких сигнатурах — и вы начинаете копировать одно и то же снова и снова. Это утомительно и опасно: поправишь форму в одном месте, забудешь в другом — и типы разойдутся.

Решение — дать форме имя и переиспользовать его. Для этого есть два инструмента: интерфейсы (interface) и псевдонимы типов (type). Оба позволяют написать форму один раз и ссылаться на неё по имени; чем они отличаются — на странице interface или type.

Дальше. Поля бывают необязательными или защищёнными от записи — про ? и readonly читайте в разделе Опциональные и readonly.

Кратко

  • Тип объекта описывает форму: какие поля есть и какого они типа. Пишется инлайном: let user: { name: string; age: number }.
  • Свойства разделяют точкой с запятой, запятой или переносом строки — это равнозначно.
  • Значением поля может быть снова объект — так получаются вложенные типы.
  • TypeScript ловит отсутствующее поле, поле неверного типа и лишнее поле у литерала (excess property check).
  • Тип ≠ значение: слева от = — форма (исчезает после компиляции), справа — данные (живут в рантайме).
  • Тип объекта удобно ставить параметром и результатом функции.
  • Повторяющиеся инлайн-формы выносят в имя — интерфейс или псевдоним типа.
