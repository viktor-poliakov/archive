TypeScript/Объекты и интерфейсы/Интерфейсы

Интерфейсы

Интерфейс — это способ дать имя форме объекта и потом переиспользовать её сколько угодно раз. Вместо того чтобы каждый раз расписывать, из каких свойств состоит объект, вы один раз описываете образец под именем — и дальше просто ссылаетесь на это имя. В этом уроке разберём, как объявлять интерфейсы, ставить их как аннотации, писать методы, необязательные свойства и расширять один интерфейс другим.

Зачем нужны интерфейсы

На прошлой странице мы описывали форму объекта прямо на месте — инлайново: { name: string; age: number }. Это удобно для одного раза, но как только одна и та же форма нужна в нескольких местах, её приходится копировать.

Проблема не в объёме букв, а в том, что форма размазана по коду. Захотели добавить поле email — надо найти и поправить все копии. Забыли одну — получили рассинхрон и странные ошибки. Интерфейс решает это: форма живёт в одном месте под именем, а везде дальше стоит только имя.

Аналогия. Интерфейс — как бланк или техпаспорт: один утверждённый образец, по которому проверяют множество экземпляров. Бланк заявления в учреждении один, а заполненных заявлений — тысячи, и каждое сверяют с этим единственным образцом. Меняется правило — правят один бланк, а не каждую заполненную бумажку.

Синтаксис объявления

Интерфейс объявляют ключевым словом interface, за ним имя и в фигурных скобках — список свойств с их типами. Важная деталь: знака = здесь нет (в этом отличие от псевдонима type, где равно ставится).

Свойства внутри разделяют точкой с запятой или просто переносом строки — оба варианта работают. Имя интерфейса по соглашению пишут с большой буквы: User, Product, OrderItem.

Важно. Интерфейс существует только на этапе проверки типов. В собранном JavaScript от него не остаётся ни строчки — он не создаёт объектов и ничего не «весит» в рантайме. Это чистое описание формы для компилятора.

Как ставить интерфейс: аннотация

Готовое имя интерфейса ставят там же, где обычную аннотацию типа — после двоеточия. Работает и для переменных, и для параметров функций, и для типа возвращаемого значения.

Обратите внимание: имя User заменило всю форму { name: string; age: number }. Читать код стало проще, а форму теперь можно менять в одном месте. Подробнее про сами типы объектов — на странице Типы объектов.

Методы в интерфейсе

Метод — это свойство, значением которого является функция. Записать его в интерфейсе можно двумя способами, и они означают практически одно и то же.

  • Свойство-функция: log: (message: string) => void — имя, двоеточие и тип-стрелка.
  • Сокращённая запись метода: warn(message: string): void — имя со скобками, как объявление функции.

Почему так. Оба варианта задают метод, и на практике их часто используют как взаимозаменяемые. Тонкая разница проявляется лишь в очень строгих режимах при проверке совместимости параметров; для начала считайте их эквивалентными и выбирайте тот, что читается приятнее.

Необязательные и readonly свойства

Не каждое свойство обязано присутствовать, и не каждое можно менять. Два коротких модификатора закрывают эти случаи:

  • имя?: тип — свойство необязательное, объект может его не иметь.
  • readonly имя: тип — свойство только для чтения: задали при создании и больше не меняем.

Здесь мы касаемся темы лишь поверхностно. Подробный разбор — что считается «отсутствием» свойства, как readonly взаимодействует с массивами и вложенными объектами — на отдельной странице Опциональные и readonly.

Расширение: extends

Часто одна форма — это другая форма плюс несколько новых свойств. Чтобы не переписывать общее заново, интерфейс можно расширить ключевым словом extends: он унаследует все свойства родителя и добавит свои.

Наследовать можно и от нескольких интерфейсов сразу, и выстраивать целые цепочки. Все тонкости — на странице Расширение и наследование. А если форму объекта должен гарантировать класс, для этого есть ключевое слово implements.

Практика: от формы до ошибки

Соберём всё вместе. Опишем интерфейс Book, создадим по нему пару объектов и передадим их в функцию, которая ждёт именно Book.

А теперь посмотрим, что произойдёт, если объект не соответствует форме. TypeScript ловит все три типичные ошибки ещё до запуска: нет обязательного свойства, неверный тип свойства и лишнее поле в литерале.

Именно ради этого интерфейсы и нужны: форма описана один раз, и любое отклонение от неё подсвечивается сразу. Сравнение interface и type — на странице interface или type.

Важно. Проверка на лишнее поле (bad3) срабатывает только для объектных литералов, написанных прямо в присваивании. Если тот же объект сначала положить в переменную, а потом присвоить — лишнее поле пройдёт. Это отдельный нюанс структурной типизации, разобранный в теме Нюансы и подводные камни.

Кратко

  • Интерфейс даёт имя форме объекта, чтобы описать её один раз и переиспользовать: interface User { name: string; age: number }.
  • В объявлении нет знака = (в отличие от type); имя пишут с большой буквы.
  • Ставится как обычная аннотация: let u: User = {…}, function greet(u: User).
  • Метод записывают двумя способами: свойством-функцией log: () => void или сокращённо warn(): void.
  • имя? — необязательное свойство, readonly имя — только для чтения (подробнее — Опциональные и readonly).
  • extends расширяет один интерфейс другим — наследует свойства и добавляет новые (подробнее — Расширение).
  • Если объект не соответствует форме, TypeScript покажет ошибку ещё до запуска.
