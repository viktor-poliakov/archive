Нюансы и подводные камни

Типы объектов, интерфейсы и type устроены логично, но в них есть несколько мест, где поведение неожиданно для новичка — да и опытного порой ставит в тупик. Ниже — коллекция самых частых «граблей»: почему лишнее поле то ловится, то нет; почему readonly не всегда спасает; чем name?: T отличается от name: T | undefined. Для каждого пункта — короткое объяснение, пример и как сделать правильно.

1. Проверка лишних свойств срабатывает не всегда

Если вы присваиваете переменной с известным типом объектный литерал (то есть пишете { ... } прямо на месте) и в нём есть поле, которого нет в целевом типе, — TypeScript ругается. Это называют excess property check — проверка на лишние свойства.

Казалось бы, лишнее поле всегда ошибка. Но стоит положить тот же объект в промежуточную переменную — и проверка исчезает: значение проходит молча. Это классический сюрприз.

Почему так. Если вы прямо тут создаёте объект и вписываете в него поле, которого нет в типе, — это почти наверняка опечатка, и компилятор предупреждает. А когда объект уже лежит в переменной, действует обычное структурное правило: объект с лишними полями всё равно подходит. Подробнее про сами типы объектов — на странице Типы объектов.

2. Структурная («утиная») типизация

TypeScript сравнивает типы объектов не по имени, а по форме. Действует принцип «если нечто крякает как утка — значит, утка»: объект подходит под тип, если у него есть все нужные поля правильных типов, даже если формально он «другого сорта».

Аналогия. Розетке всё равно, какой на вилке бренд — важно, чтобы совпали форма и штырьки. Так и здесь: имя класса или интерфейса не проверяется, проверяется набор полей. Это удобно, но иногда «случайно совместимые» объекты проходят туда, куда вы их не планировали пускать.

3. name?: T — это не name: T | undefined

Оба варианта разрешают значению «отсутствовать», но по-разному, и путаница тут частая. Знак вопроса ? делает сам ключ необязательным — его можно вообще не писать. А | undefined оставляет ключ обязательным, просто разрешает в нём значение undefined.

Запись Ключ можно опустить? { } подходит?
name?: string да, ключ необязателен ✅ да
name: string | undefined нет, ключ обязателен ❌ нет, нужен ключ

Подробный разбор опциональных полей — на странице Опциональные и readonly.

4. readonly поверхностный и живёт только при компиляции

Модификатор readonly запрещает переприсваивать поле. Но у него два важных ограничения. Первое: он поверхностный (shallow) — защищена только сама ссылка, а содержимое вложенного массива или объекта менять можно. Второе: он существует только на этапе компиляции — в собранном JavaScript этого ограничения нет.

Важно. Если нужна по-настоящему неизменяемая структура, одного readonly на поле мало — придётся помечать readonly и вложенные типы, использовать readonly T[] для массивов или замораживать объект в рантайме через Object.freeze. Про модификатор — на странице Опциональные и readonly.

5. Индексная сигнатура прячет опечатки

Индексная сигнатура вида [key: string]: number говорит: «любой строковый ключ даёт число». Удобно для словарей, но есть побочка: обращение по несуществующему ключу не считается ошибкой — вы молча получаете undefined, а компилятор уверен, что там число.

Лечится флагом компилятора noUncheckedIndexedAccess: с ним результат доступа по индексу становится number | undefined, и TypeScript заставит вас проверить значение перед использованием. Подробнее про сами сигнатуры — Индексные сигнатуры.

6. Коварные типы: object, Object и {}

Тип {} выглядит как «пустой объект», но означает совсем другое — «почти что угодно, кроме null и undefined». В него влезут и число, и строка, и boolean. То же касается заглавного Object. Если нужен именно объект, есть отдельный тип object (со строчной буквы) — он исключает примитивы.

Важно. Аннотация : {} почти никогда не то, что вы имели в виду: она не запрещает передать число или строку. Хотите «любой объект» — пишите object; хотите «объект с произвольными полями» — берите индексную сигнатуру или Record<string, unknown>.

7. Интерфейсы молча сливаются

Если объявить два интерфейса с одинаковым именем в одной области видимости, TypeScript не выдаст ошибку, а незаметно объединит их поля в один (это называется declaration merging). Иногда это специально используют (например, чтобы дополнить типы библиотеки), но чаще это ловушка: вы случайно назвали свой интерфейс как уже существующий — и он «оброс» чужими полями.

У псевдонимов типов type такого поведения нет: два одинаковых type — это ошибка-дубликат. Это одно из отличий, разобранных на странице interface или type.

8. Доступ к опциональному полю без проверки

При включённом strict обращаться к вложенному свойству опционального поля напрямую нельзя — оно может оказаться undefined, и программа упадёт в рантайме. TypeScript ловит это заранее. Правильный путь — сначала проверить поле или использовать опциональную цепочку ?..

Аналогия. Прежде чем достать письмо из почтового ящика, стоит убедиться, что ящик вообще есть. ?. делает это автоматически: если «ящика» (значения) нет, вся цепочка спокойно вернёт undefined вместо падения.

Кратко

  • Лишние поля ловятся у объектного литерала на месте (excess property check), но проходят, если объект пришёл через переменную.
  • Структурная типизация: совместимость по форме (набору полей), а не по имени типа или класса.
  • name?: T разрешает отсутствие ключа; name: T | undefined требует ключ, но допускает значение undefined.
  • readonly поверхностный (вложенное содержимое не защищено) и действует только при компиляции.
  • Индексная сигнатура прячет опечатки — доступ по любому ключу «валиден»; включайте noUncheckedIndexedAccess.
  • {} и Object — это «почти что угодно, кроме null/undefined», а не «пустой объект»; для объекта берите object или Record.
  • Одноимённые интерфейсы сливаются (declaration merging); у type такого нет.
  • К опциональному полю обращайтесь через проверку или ?. — иначе при strict будет ошибка.
