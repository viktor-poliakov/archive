Опциональные и readonly-свойства

У свойства объекта, кроме имени и типа, бывают ещё два «значка»-модификатора. Знак вопроса ? говорит «этого поля может и не быть», а слово readonly — «поле можно задать один раз, но потом не менять». Оба значка помогают точнее описать форму объекта и заранее ловить ошибки. В этом уроке разберём каждый по отдельности, покажем частые ловушки и соберём всё на практике.

Опциональное свойство ?

Поставьте знак вопроса сразу после имени свойства — name?: string — и поле станет необязательным. Это значит, что в объекте такого типа поля может не быть вообще: ни ключа, ни значения. А объявленные без ? поля остаются обязательными — забыть их нельзя.

Аналогия. Представьте бумажную анкету. Поле «Фамилия» обязательное — без него анкету не примут. А поле «Отчество (при наличии)» помечено звёздочкой «необязательно»: можно вписать, а можно оставить пустым, и анкета всё равно действительна. Знак ? — это ровно та самая пометка «можно не заполнять».

Перед использованием — проверьте

Тип необязательного поля фактически становится «значение ИЛИ undefined». Поэтому в строгом режиме (strict) TypeScript не даст сразу обратиться к такому полю или вызвать у него метод — вдруг его нет. Сначала нужно проверить, что значение действительно есть.

Про то, как компилятор «сужает» тип после проверки if (u.name) и убирает из него undefined, подробно рассказано на странице Объединения и сужение.

name?: string против name: string | undefined

Эти две записи похожи, но отличаются в одном важном моменте — обязателен ли сам ключ. С ? ключ можно вообще не писать. А с | undefined без ? ключ обязателен: значение может быть undefined, но передать сам ключ придётся явно.

Что можно name?: string name: string | undefined
Опустить ключ целиком ✅ да ❌ нет, ключ нужен
Передать значение undefined ✅ да ✅ да
Передать строку ✅ да ✅ да

Важно. На практике почти всегда нужен именно ?: он описывает «поля может не быть». Форма string | undefined без ? встречается редко — когда вы хотите заставить вызывающий код осознанно передать «пусто», а не забыть про поле по невнимательности.

Свойство readonly — только для чтения

Слово readonly перед именем свойства разрешает задать значение один раз — при создании объекта — и запрещает менять его потом. Читать такое поле можно сколько угодно, а вот присвоить ему новое значение компилятор не даст.

Аналогия. Это как серийный номер, выгравированный на корпусе устройства. В момент выпуска его наносят — а после стереть и написать другой уже нельзя. Поле readonly id ведёт себя так же: значение «выгравировано» при создании объекта.

Честно: readonly живёт только в типах

Скажем прямо: readonly — это проверка только на этапе компиляции. В готовом JavaScript-коде такое поле остаётся самым обычным свойством, никакой реальной защиты у него нет. Компилятор просто подсвечивает попытки присваивания в вашем коде, но обойти проверку типов (например, через as any) — и значение изменится во время выполнения.

Почему так. TypeScript исчезает после компиляции — в браузере или Node.js исполняется чистый JavaScript, который про readonly ничего не знает. Если нужна настоящая неизменяемость во время выполнения, применяйте Object.freeze() — это уже механизм самого JavaScript.

Массивы только для чтения

readonly работает и с массивами. Есть две равнозначные записи: readonly number[] и ReadonlyArray<number> — это одно и то же. У такого массива можно читать элементы и длину, но нельзя менять: методы вроде push, pop, sort и присваивание по индексу становятся недоступны.

Это удобно для функций, которым массив нужен только «на чтение»: пометив параметр как readonly number[], вы гарантируете, что функция случайно не испортит переданные данные.

Оба значка вместе и практика

Модификаторы не спорят друг с другом — свойство может быть одновременно и readonly, и необязательным: readonly nickname?: string. Читается как «поле можно не указывать, но если указали — менять его нельзя».

Практика 1: объект конфигурации

Типичный случай для ? — настройки, где часть полей обязательна, а часть имеет значения по умолчанию. Необязательные поля подстраховываем оператором ??.

Практика 2: сущность с readonly id

Типичный случай для readonly — идентификатор записи в базе. Его выдают один раз при создании, и по логике приложения он не должен меняться. Пометка readonly защищает от случайной перезаписи прямо в редакторе.

Когда таких полей много и хочется сделать все свойства необязательными или все только для чтения разом, пригодятся вспомогательные типы Partial и Readonly — про них есть отдельная страница Partial, Required, Readonly. А базовые способы описывать форму объекта разобраны в разделах Типы объектов и Интерфейсы.

Кратко

  • ? — необязательное свойство: поля может не быть совсем. Тип поля фактически «значение | undefined», поэтому перед использованием его нужно проверить.
  • name?: string — ключ можно опустить; name: string | undefined — ключ обязателен, но его значением может быть undefined.
  • readonly — только чтение: значение задаётся при создании и не переписывается потом.
  • readonly проверяется только при компиляции; в рантайме поле обычное. Для настоящей неизменяемости — Object.freeze().
  • Массивы: readonly number[] и ReadonlyArray<number> запрещают push, sort и запись по индексу.
  • Значки комбинируются: readonly name?: string. Массово навесить их помогают Partial и Readonly.
