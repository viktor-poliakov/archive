TypeScript/Объекты и интерфейсы/interface или type

interface или type

Это, наверное, самый частый вопрос новичка в TypeScript: описывать объект через interface или через type? Хорошая новость — для обычной формы объекта они почти взаимозаменяемы. Разница появляется в деталях: у type шире набор возможностей, а у interface есть один уникальный трюк. Разберём всё по полочкам и в конце дадим простое правило выбора.

Для формы объекта они почти близнецы

Если вам нужно просто описать «объект с такими-то полями», оба инструмента дадут один и тот же результат. Синтаксис отличается мелочью: interface ставит фигурные скобки сразу после имени, а type — это присваивание, поэтому у него есть знак =.

Аналогия. Представьте два бланка анкеты на одного и того же человека: один напечатан на бумаге A4, другой — на бумаге A5. Поля в анкете одинаковые, отличается только «рамка» вокруг. Так и здесь: interface UserI и type UserT описывают одного и того же пользователя, разной остаётся лишь форма записи. Проверять объекты и подсказывать ошибки TypeScript будет абсолютно одинаково.

Про формы объектов подробно — на страницах Интерфейсы и Псевдонимы типов type. Здесь мы сравниваем их между собой.

Таблица различий

Совпадают они в главном — в описании объектов. Расходятся в том, что каждый умеет помимо этого. Вот полная картина:

Возможность interface type
Форма объекта да — основное назначение да — так же удобно
Объединения (union) нет да — 'a' | 'b'
Псевдоним примитива / кортеж / функция нет да — type Id = number
Расширение да — через extends да — через &
Слияние объявлений (declaration merging) да — уникальная фишка нет — имя занять нельзя дважды

Если коротко: type «умеет больше разных типов», а interface выигрывает в одном специальном сценарии — слиянии объявлений. Ниже разберём каждую строку с примерами.

Что умеет только type

type — это буквально «дать имя любому типу». Поэтому он справляется с тем, что вообще не является объектом.

Объединения (union)

Самый частый повод взять именно type — перечислить набор допустимых значений через |. Про них подробно на странице Объединения (union).

Псевдоним примитива, кортеж, функция

Хотите дать говорящее имя обычному числу, описать пару координат (примитивы и кортежи) или тип функции — это работа для type.

Вычисляемые типы

Более продвинутое: типы, которые «считаются» из других — mapped-типы (пройтись по ключам), conditional-типы (ветвление) и шаблонные литеральные типы. Всё это только через type. На старте их знать необязательно, но полезно понимать, что именно тут interface бессилен.

Важно. Многие удобные утилиты вроде Partial и Required или Record внутри устроены как раз на таких вычисляемых типах — поэтому они и живут в мире type.

Что умеет только interface: слияние объявлений

Здесь у interface козырь. С type имя типа можно занять только один раз — второе объявление с тем же именем будет ошибкой:

А вот два interface с одинаковым именем TypeScript молча склеит в один. Это и называется declaration merging — слияние объявлений.

Аналогия. type — как подпись на двери кабинета: табличку с именем «Box» можно повесить лишь одну, вторая такая же — конфликт. А interface — как общий список дел на холодильнике: каждый может дописать свою строчку, и в итоге получается один общий список. Оба листка «Box» превращаются в единый.

Зачем это нужно на практике? Чтобы дополнить чужой тип, к исходникам которого у вас нет доступа — например, добавить своё поле в глобальный Window или расширить типы сторонней библиотеки.

Расширение: extends против &

Расширять умеют оба, просто по-разному. У interface для этого есть ключевое слово extends — выглядит как классическое наследование.

У type роль «наследования» играет пересечение & (intersection): оно складывает несколько типов-объектов в один, где обязательны все поля сразу.

Приятно, что их можно смешивать: interface способен расширить объект, объявленный через type, и наоборот. Так что выбор одного стиля не запирает вас в нём навсегда.

Почему так. extends у интерфейсов даёт чуть более понятные сообщения об ошибках (компилятор знает «родителя» по имени) и проверяет совместимость полей строже. & же — универсальнее, работает с любыми типами, но при конфликте полей может тихо получиться never. Полный разбор — на странице Расширение и наследование.

Что же выбрать на практике

Строгого закона нет — оба варианта корректны, и во многих командах это просто вопрос договорённости о стиле. Но есть разумное правило по умолчанию, которое избавит от лишних раздумий.

Задача Удобнее
Форма объекта, модель данных, публичный APIinterface
Объединение значений ('a' | 'b')type
Имя для примитива, кортежа, функцииtype
Mapped / conditional / шаблонные типыtype
Нужно дополнять тип из библиотеки (merging)interface

Важно. Это рекомендация, а не догма. Некоторые команды пишут вообще всё через type ради единообразия — и это нормально. Главное — выбрать один стиль и держаться его в рамках проекта.

Кратко

  • Для формы объекта interface и type почти взаимозаменяемы — разница лишь в синтаксисе (type с =).
  • Только type: объединения (union), псевдонимы примитивов, кортежи, типы функций, вычисляемые типы (mapped / conditional / шаблонные).
  • Только interface: слияние объявлений (declaration merging) — два одноимённых интерфейса склеиваются; удобно дополнять чужие типы.
  • Расширение: у interface — через extends, у type — через &; их можно смешивать.
  • Правило выбора: объекты и публичные API — interface; union, псевдонимы и сложные типы — type. Это стиль, а не жёсткий закон.
  • Дальше полезно посмотреть Расширение и наследование и Нюансы и подводные камни.
