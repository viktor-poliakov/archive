Хранилище: нюансы и подводные камни

Браузерное хранилище выглядит просто, но именно тут легко получить тихий баг или брешь в безопасности. Соберём то, обо что спотыкаются чаще всего.

Только строки

Web Storage и cookies хранят исключительно строки. Число молча станет строкой, и арифметика превратится в склейку; объект без JSON — в бесполезное '[object Object]':

Лимит места

Web Storage невелик — обычно около 5–10 МБ на сайт. Если попытаться записать больше, браузер бросает исключение QuotaExceededError, поэтому запись потенциально больших данных стоит оборачивать в try / catch:

Нужно хранить действительно много — это сигнал перейти на IndexedDB.

Web Storage синхронный

localStorage и sessionStorage работают синхронно — каждая операция блокирует главный поток, пока не завершится. Для мелочи это незаметно, но большие объёмы данных подвесят интерфейс. Тяжёлые данные — снова повод взять асинхронный IndexedDB (см. также Event Loop — почему блокировка потока «замораживает» страницу).

Безопасность

Всё в localStorage доступно любому скрипту на странице. Если на сайт попадёт чужой код (атака XSS), он прочитает оттуда всё — включая токены. Поэтому секреты в localStorage не хранят:

Кратко

  • Только строки: числа приводятся к строке, объекты — через JSON.stringify/JSON.parse.
  • Лимит ~5–10 МБ у Web Storage; переполнение → QuotaExceededError.
  • Web Storage синхронный — большие операции подвешивают UI; для объёмов берите IndexedDB.
  • Не храните секреты в localStorage — его читает любой скрипт (XSS); токены — в HttpOnly-cookie.
  • Всё изолировано по сайту: Web Storage и IndexedDB — по origin (протокол + домен + порт), cookies — по домену и пути.
