Протокол итерации

Почему for...of одинаково хорошо перебирает массив, строку и Map, но спотыкается об обычный объект? Потому что у них есть общая договорённость, как выдавать элементы по одному — её называют протоколом итерации. Разберёмся, из чего он состоит: это ключ ко всей теме.

Аналогия — дозатор. Представьте автомат, который по нажатию кнопки выдаёт по одному стаканчику, пока они не кончатся, а потом честно говорит: «пусто». Всё, что умеет так «выдавать по одному», в JavaScript и есть итерируемое.

Два понятия: iterable и iterator

В основе — два связанных объекта:

  • iterable (итерируемое) — то, что можно перебрать. У него есть метод с особым ключом [Symbol.iterator](), который выдаёт итератор.
  • iterator (итератор) — сам «механизм выдачи». У него есть метод next(), и каждый его вызов возвращает объект { value, done }: value — очередной элемент, done — закончились ли элементы (false — ещё есть, true — конец).

Что итерируемо, а что нет

Итерируемы «из коробки» массивы, строки, Set, Map и некоторые другие. А вот обычный объект — нет (у него просто нет [Symbol.iterator]), поэтому for...of по нему бросает ошибку — это частая неожиданность для новичков.

Для перебора обычного объекта берут Object.keys/values/entries — см. Объекты: перебор свойств. А строка итерируется по «настоящим» символам (см. Строки: Юникод).

Как это работает вручную

Чтобы увидеть протокол «вживую», возьмём итератор у массива сами и покрутим next() руками. Обычно это за нас делает for...of, но под капотом происходит ровно это:

Кратко

  • iterable — есть метод [Symbol.iterator](), возвращающий итератор.
  • iterator — есть next(), возвращающий { value, done }.
  • done: false — ещё есть элементы; done: true — перебор окончен.
  • Обычный объект не итерируем — for...of по нему бросает TypeError.
