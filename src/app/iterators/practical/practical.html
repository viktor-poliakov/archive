Практика и асинхронные итераторы

Соберём всё вместе на реальных задачах: обход дерева, ленивый конвейер обработки и асинхронный перебор данных с сервера. В конце — типичные подводные камни.

Обход дерева

Рекурсивный генератор с yield* элегантно «разворачивает» вложенную структуру в плоскую последовательность. Каждый узел отдаёт своё значение, а затем делегирует обход детям — и так до самых листьев.

Ленивый конвейер

Привычная цепочка arr.map(...).filter(...) на массивах делает два полных прохода: сначала .map() обходит весь массив и строит промежуточный массив ([10, 20, 30, 40]), а затем .filter() обходит уже его и строит ещё один. Получается лишний массив в памяти и лишний проход.

Если написать map и filter как генераторы, промежуточные массивы не создаются вообще: каждый элемент проходит всю цепочку целиком и только потом берётся следующий.

Когда это реально выгодно. На крошечном массиве, как здесь, разницы нет — обычный .map().filter() проще и читабельнее, его и берите. Выгода генераторов появляется на больших данных (не тратим память на промежуточные массивы) и особенно на бесконечных источниках: чтобы применить обычные .map().filter(), сначала нужен массив, а превратить бесконечный генератор в массив ([...naturals()]) нельзя — зависнет. Ленивый конвейер работает прямо на генераторе и его можно оборвать в любой момент (например, take(n)).

Асинхронные итераторы

А что, если следующий элемент нужно дождаться — например, подгружать с сервера страницу за страницей? Для этого есть асинхронная версия протокола: async function* (внутри её yield можно await-ить), а перебирают такой источник циклом for await...of.

Здесь важны две вещи. Во-первых, вызов fetchPages('/api/list') ничего не скачивает сразу — он возвращает асинхронный итератор, готовый выдавать страницы по запросу. Во-вторых, for await...of на каждом витке просит у него следующую страницу, ждёт ответа сервера и кладёт в items её порцию — массив записей одной страницы. Поэтому console.log(items) печатает не всё сразу, а по одному массиву за страницу: сперва ['a', 'b'], затем ['c', 'd'], и цикл завершается, когда nextUrl станет null.

Под капотом это тот же протокол, только next() возвращает промис с { value, done }. Ровно этот паттерн вы уже видели «вручную» — цикл reader.read() → { done, value } при чтении потока в разделе Файлы. Про сам await и асинхронность — async/await и Event Loop.

Подводные камни

  • Итератор одноразовый. Пройдя итератор до конца, его нельзя перебрать снова — он «исчерпан». Поэтому for...of обычно применяют к iterable (массиву, объекту с [Symbol.iterator], генераторной функции), который каждый раз выдаёт свежий итератор.
  • iterable ≠ iterator. Генераторная функция gen — не то же, что генератор gen(): перебирать нужно результат вызова.
  • Бесконечный + spread = зависание. [...infinite()] и Array.from пытаются взять все элементы — ограничивайте (take(n)).
  • Генератор ленив. Вызов генераторной функции не выполняет её тело — оно начнёт работать только с первого next().

Кратко

  • Обход дерева — рекурсивный генератор + yield*.
  • Ленивый конвейер — map/filter генераторами, поэлементно, без промежуточных массивов; выгода — на больших и бесконечных данных.
  • Асинхронно — async function* + for await...of (next() возвращает промис).
  • Помните: итератор одноразовый; бесконечный не разворачивайте целиком; генератор ленив.
