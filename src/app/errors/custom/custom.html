Свои классы ошибок

Встроенных типов (TypeError, RangeError…) хватает не всегда: у приложения свои виды проблем — «не прошла валидация», «нет доступа», «не найдено». Для них заводят собственные классы ошибок, наследуя от Error.

Аналогия — разные бланки для разных проблем. Вместо одной безликой «жалобы» вы заводите отдельные формы: «ошибка ввода», «нет прав», «не найдено». Обработчик, взглянув на бланк, сразу понимает, что случилось, и знает, что делать.

class X extends Error

Свой класс ошибки — это класс, унаследованный от Error (см. Наследование). В конструкторе обязательно вызывают super(message) и задают this.name (иначе он останется 'Error'). Заодно можно добавить любые свои поля — например, какое поле формы не прошло проверку.

Различаем по instanceof

Главная польза своих классов — их можно отличать от чужих ошибок в общем catch. Свою ошибку обрабатываем осмысленно, а всё неожиданное — пробрасываем дальше (throw err), чтобы не «проглотить» настоящий баг.

Иерархия ошибок

Когда своих ошибок много, их удобно объединить общим базовым классом (например, AppError). Тогда одним instanceof AppError можно поймать все ошибки приложения разом, отделив их от системных.

Живой пример: класс ApiError extends Error для сетевых ошибок — в разделе REST API: обработка ошибок.

Практические примеры

Валидация формы

Классический сценарий: проверяем поля формы и бросаем ValidationError с указанием, какое именно поле не прошло. Обработчик ловит ошибку и подсвечивает нужное поле — сообщение попадает точно по адресу, а не в общий алерт.

Слой API: разная реакция на разные ошибки

Функция загрузки превращает коды ответа в типизированные ошибки (NotFoundError, ForbiddenError), а вызывающий код по типу решает, что показать: страницу «не найдено», редирект на вход или проброс дальше. Логика «что случилось» отделена от логики «как реагировать».

Так удобно строить обработку по слоям: нижний слой (запрос) знает, какая ошибка произошла, а верхний (UI) — как на неё отреагировать. Про сетевые ошибки и fetch — в разделе REST API: обработка ошибок.

Кратко

  • class X extends Error — свой тип ошибки; в конструкторе super(message) и this.name = '...'.
  • Свои поля (например field, code) — добавляйте по необходимости.
  • Различайте через instanceof; чужие ошибки пробрасывайте дальше.
  • Базовый класс (AppError) — «зонтик», чтобы ловить все свои ошибки одним instanceof.
